## Goal

Implement task 121 for GitHub issue #375: add a viewport-only left-double-click
frame-all camera gesture in `rge-editor-shell`. A qualifying left double-click
over the Viewport tab body must reframe `EditorShell::editor_camera` through the
existing `reset_camera()` path, which already uses `current_scene_bounds()` plus
`isometric_camera_for_bounds(min, max)` and falls back to
`EditorCameraState::default()` when no scene bounds are frameable. This is the
smallest next camera/navigation slice after viewport wheel zoom, right-button
orbit, and middle-button pan; it must not create a new command, menu item,
accelerator, route, CAD/action mutation path, or UI-host surface.

  `crates/editor-shell/src/lifecycle/viewport_navigation.rs`, alongside
  `ViewportOrbitDrag` and `ViewportPanDrag`, with explicit time and movement
  thresholds and tests that can drive it deterministically.
- Lifecycle routing in `crates/editor-shell/src/lifecycle/mod.rs` so a
  qualifying second left press over the Viewport tab body calls the existing
  `EditorShell::reset_camera()` path. The first single left press must still
  call `handle_left_click()` when `should_fire_face_pick(...)` allows it.
- Focused tests in `crates/editor-shell/src/lifecycle/tests.rs` covering
  viewport frame-all behavior, empty/default fallback, outside/no-cursor
  no-ops, single-click face-pick preservation, and time/movement threshold
  no-frame behavior.
- Normal ISSUE-375 bookkeeping only if implementation completes: task/status
  history updates limited to `.ai/dispatch.tasks.md`, `Status.md`,
  `HANDOFF.md`, `plans/BASELINE.md`, and `change.md`, plus the executor's own
  ISSUE-375 handoff/log artifacts.

## Acceptance Criteria

- A left double-click whose second press lands over the current Viewport tab
  body and has a present finite `cursor_pos` reframes `self.editor_camera` by
  reusing the same `reset_camera()` behavior as the View menu/Home command:
  finite non-empty scene bounds produce the framed isometric camera, and
  `None` bounds fall back to `EditorCameraState::default()`.
- Double-click detection is private to `lifecycle/viewport_navigation.rs`,
  parameterized by a within-window time threshold and maximum pointer-movement
  threshold, and resets on non-left buttons, missing/non-finite cursor data,
  outside-viewport clicks, or clicks outside either threshold.
- The first single left-click still routes to `handle_left_click()` for
  face-pick exactly as today. The frame-all gesture fires only on the qualifying
  second click; `should_fire_face_pick(...)` single-click behavior remains
  unchanged.
- The gesture is both viewport-gated and cursor-gated. Double-clicks outside
  the Viewport tab body, with no cursor, with no viewport/egui hit-test state,
  or with non-finite cursor coordinates are no-ops and must not panic.
- No `Command` variant, menu entry, accelerator, command-palette behavior,
  `route_menu_command` / `MenuCommandHandoff` routing, host/UI code, CAD graph
  or projection mutation, editor-action/CommandBus mutation, undo/dirty policy,
  Cargo metadata, render-path ownership, or broader input-controller work is
  introduced.
- Right-button orbit, middle-button pan, mouse-wheel zoom, View reset/zoom
  commands, menu/palette routing, keyboard accelerator routing, and existing
  face-pick tests remain passing.
- Only the paths listed in `MAY edit` and generated ISSUE-375 artifacts listed
  in `MAY add new files` change.
