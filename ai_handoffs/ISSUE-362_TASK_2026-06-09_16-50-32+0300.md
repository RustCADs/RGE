## Deliverables

- A host-local diagnostics affordance that is visible only when
  `ProjectedMainMenu.conflicts` is non-empty and opens an inspectable egui
  window or similarly persistent host surface.
- Displayed conflict rows sourced only from existing
  `ProjectedMainMenu.conflicts`, preserving registry-provided shortcut display
  strings and entry-id order. More than one conflict must render
  deterministically.
- Focused host tests covering conflict row projection/display order, multiple
  conflicts, default-menu no-conflict behavior, and read-only UI state
  behavior.
- Optional status/task documentation updates only if they accurately record the
  completed issue and verification results.
- An `EXECUTION_REPORT` for ISSUE-362 documenting changed files, verification
  output, and any deviations.

## Acceptance Criteria

- The diagnostics surface replaces or augments the current inline
  `"Shortcut Conflicts"` dropdown with an inspectable host-local surface when
  conflicts exist.
- Every displayed row comes from `ProjectedMainMenu.conflicts`; no hard-coded
  shortcut table, second registry, or duplicate conflict detector is added.
- Shortcut display strings and conflicting entry ids remain in the order
  provided by the projected menu data.
- `MenuRegistry::resolve` semantics are preserved: the first registered
  shortcut winner is unchanged, conflicts remain non-fatal diagnostics, and no
  default-menu conflict is introduced.
- Opening, closing, or inspecting the diagnostics surface does not enqueue a
  `MenuCommandHandoff` command, does not alter command-palette recents, pins,
  filter, or selection, does not toggle keyboard-shortcuts help, and does not
  mutate editor-shell state.
- Existing command-palette, pinned/recent, shortcut-help, menu projection, and
  `MenuCommandHandoff` behavior outside the conflict diagnostics surface remains
  unchanged.
