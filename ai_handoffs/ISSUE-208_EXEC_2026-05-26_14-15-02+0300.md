### Docs

- `AI_DISPATCH_AUTOMATION.md`:
  - Table of contents: added entry `18. Delegated-human auto-publish policy`.
  - §7.1 (Unattended operation): added a cross-reference sentence in the
    `Invoke-AiDispatchAuto.ps1` paragraph pointing at §18 and reaffirming
    `-PublishMode branch` as the default and safest mode. Did not change
    the surrounding description of the three unattended-operation layers.
  - §18 (new section): documents the delegated-human auto-publish policy
    in seven subsections:
    - §18.1 — default mode and the meaning of delegated-human
      authorization. States `-PublishMode branch` as default/safest;
      `-PublishMode main` as explicit, bounded, opt-in human authorization
      for a single named batch; defines "indefinite" operation as repeated
      bounded batches, never removal of caps.
    - §18.2 — allowed publish surfaces. Distinguishes lower-risk
      (docs / test-only) surfaces from higher-risk (production Rust source,
      scripts, workflows, schemas, scheduler configuration, dependency and
      Cargo metadata, the canonical verification gate, the autonomous task
      brief) surfaces.
    - §18.3 — cap rules. Reaffirms finite `-MaxAutonomousTasks`, finite
      `-MaxPlanRevisions`, finite `-MaxCorrectionRounds`, the queue
      single-run lock, and the bounded-batches definition of indefinite
      operation.
    - §18.4 — stop conditions. Enumerates failed canonical verification,
      failed/blocked Codex control, failed/blocked Claude plan gate, scope
      drift, dirty or unexpected diffs, missing audit artifacts, branch /
      base mismatch, auth or remote failure, scheduler / default-mode
      changes, and any attempted edit to forbidden surfaces. States that a
      halt under any of these must not fast-forward `main` and must not
      auto-advance to the next batch task without fresh authorization.
    - §18.5 — rollback behavior. Covers unpushed local changes (leave on
      branch for human review; salvage allowed per §14.8 / §14.12; no
      silent overwrite) and already-published `main` updates (stop further
      batches; use normal `git revert` / PR revert / audit practice; no
      history rewrite; no force-push).
    - §18.6 — audit requirements. Requires the committed queue audit log,
      handoff packets, control verdict JSON, verification gate output,
      and full issue/branch/commit linkage per run, plus a retrievable
      authorization record (who, scope, cap, mode, review note) per batch.
    - §18.7 — what this dispatch does and does not do. Explicitly states
      that ISSUE-208 does not enable `-PublishMode main`, does not register
      / modify / run a schedule, does not edit any listed forbidden
      surface, does not change any default `-PublishMode`, and does not
      imply a standing authorization.
  - Closing italic footer left intact.
