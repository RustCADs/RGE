## Goal

Remove the empty-tick steady-state delay from `Invoke-AiDispatchAuto.ps1` for
GitHub issue #188. The auto driver currently performs two five-second primary
queue-query retries when the initial `ai-dispatch` queue check returns zero
before it reaches the cap check. This dispatch must eliminate those two
steady-state sleeps so an empty queue falls through immediately to the existing
REST issues cross-check, while preserving the safety behavior that prevents
new work selection or cap-halting on ambiguous queue state.

## Deliverables

- In `Invoke-AiDispatchAuto.ps1`, remove the two steady-state five-second
  queue-check sleeps that run when the first primary open `ai-dispatch` queue
  query returns zero before the autonomous task cap check.
- Remove the associated two-attempt primary queue retry loop rather than
  replacing the sleeps with a zero-delay or shorter-delay retry. After the
  primary queue query reports zero, the existing REST issues cross-check must
  run immediately as the fallback.
- Keep `Get-OpenQueueIssuesRest` as the immediate fallback for a zero-count
  primary queue query. The fallback must still:
  - convert REST-visible queued issues into `$openQueue` entries so queued work
    drains before the cap check,
  - print the existing REST-visible queued-work message when REST finds open
    queue issues,
  - print the existing REST-confirms-empty message when REST succeeds and finds
    no queue issues, and
  - mark queue state ambiguous and skip the tick when REST fails.
- Preserve queued-work-before-cap semantics: any open queued `ai-dispatch`
  issue found by either the primary query or the REST fallback must be drained
  by running `Invoke-AiDispatchQueue.ps1` before any autonomous task cap check
  can halt or any new issue can be selected.
- Preserve ambiguous-queue skip behavior: when the primary query returns zero
  and the REST fallback fails, the tick must still exit 0 after printing the
  existing ambiguous-queue messages, without selecting new work, without
  running the cap check, and without invoking the queue.
- Preserve the post-issue-creation visibility wait loop unchanged. The loop
  that sleeps up to twelve times after `gh issue create` so a newly filed issue
  becomes label-listable must remain functionally and textually unchanged
  except for unavoidable surrounding whitespace from this dispatch's targeted
  edit.
- Preserve existing console trace and JSONL trace behavior. Do not add,
  remove, rename, reorder, or reshape `Write-TimingTrace` events, and do not
  change the JSONL trace schema, trace file path policy, trace enablement
  policy, or trace failure behavior.
- It is acceptable to update only the nearby source comment that currently
  describes retrying the primary query, so the comment accurately says the
  empty primary result falls through to the REST issues cross-check.

## Acceptance Criteria

- The only source file changed is `Invoke-AiDispatchAuto.ps1`.
- The empty-queue path no longer waits ten seconds before the cap check: there
  is no two-iteration `Start-Sleep -Seconds 5` retry loop between
  `Write-TimingTrace "auto.queue-check: primary done (...)"` and the REST
  issues cross-check.
- When the first primary queue query returns one or more open `ai-dispatch`
  issues, behavior is unchanged: the auto driver reports the pending queue and
  proceeds to drain it without running the cap check or selecting new work.
- When the primary queue query returns zero and the REST issues fallback finds
  queued work, behavior is unchanged except for the removed steady-state wait:
  the auto driver reports the REST-visible queue, populates `$openQueue`, skips
  the cap check, and drains the queue.
- When the primary queue query returns zero and the REST issues fallback
  confirms zero queued issues, the auto driver proceeds to the existing
  autonomous task cap check immediately.
- When the primary queue query returns zero and the REST issues fallback fails,
  behavior is unchanged except for the removed steady-state wait: the auto
  driver marks queue state ambiguous, exits 0 after the existing skip messages,
  and does not run the cap check, select work, create an issue, or invoke the
  queue.
- The issue-creation visibility wait loop after `gh issue create` still sleeps
  in five-second increments for up to twelve polls and retains its existing
  success and warning output.
- Existing `Write-TimingTrace` call sites, event names, message strings, JSONL
  field names, JSONL field types, and trace enablement behavior are unchanged.
- No queue runner, loop runner, watchdog, preflight, scheduler, label, retry
  policy outside the targeted empty-queue retry removal, recovery route, Rust,
  Cargo, documentation, schema, dashboard, workflow, or test behavior changes.

## Planner Notes

The target code path is the queue-presence block in `Invoke-AiDispatchAuto.ps1`
after `auto.queue-check: primary done`. Today, when `$openQueue.Count -eq 0`,
the script enters a `for ($poll = 1; $poll -le 2 -and $openQueue.Count -eq 0;
$poll++)` loop that sleeps for five seconds before each retry. Remove that
loop and let the subsequent `if ($openQueue.Count -eq 0) { $restQueue =
Get-OpenQueueIssuesRest ... }` block run immediately.

Do not touch the separate post-create label-index wait loop later in the file;
that wait is for a newly created issue to become visible to the queue runner
and is explicitly out of scope.
