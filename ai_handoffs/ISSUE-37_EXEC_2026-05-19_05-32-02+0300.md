### Tests

- `crates/cad-projection/tests/render_adapter_smoke.rs::boolean_render_mesh_face_labels_align_with_projection_lookup`:
  new `#[test]` (Test 10) for GitHub issue #37. Builds a
  `OperatorNode::Boolean(BooleanOp::union())` graph root fed by two labeled
  Cuboid-derived inputs — a 1×1×1 `CuboidOp` on Boolean port 0, and a
  dimensionally-perturbed `CuboidOp` (depth `1.0001`, so its content-derived
  `NodeId` differs and it is not deduped) routed through a translating
  `TransformOp` (`+0.5` per axis) into the overlap zone on Boolean port 1.
  Spawns and projects the Boolean node itself as the entity root, then pins:
  - `RenderMesh.face_labels` is `Some`, non-empty, and equal in length to both
    `projected.triangle_count()` and the projected mesh's `face_labels`.
  - For every triangle index, the renderer-side `u64` label equals the
    projected `TopologyFaceId.0` at that same index — checked in triangle
    order, not by sorting or set comparison.
  - For every Boolean-root triangle index,
    `projection.brep_face_id_for_triangle(entity, tri, &world, graph.graph())`
    returns `None`, preserving the posture that Boolean is topology-changing
    for B-Rep resolution. No Boolean-root `BRepFaceId` mapping is asserted or
    introduced.
  - The projected mesh is asserted non-empty (`triangle_count() > 0`) rather
    than pinned to an exact CSG triangle count, per the task's non-goal.
- Added the `BooleanOp` import to the existing `rge_cad_core` use-list.
