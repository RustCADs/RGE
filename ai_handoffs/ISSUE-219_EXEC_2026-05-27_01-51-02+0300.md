## Candidate comparison — why the chosen follow-up

### Why a user-visible gap is real today

Rows (#8) MenuRegistry, (#9) Hierarchy panel, (#10) File→Open of a
`.rge-scene`, (#14) Save .rge-scene, and (#19) workspace selection are all
genuine user-visible gaps. Of these, (#10) is the one that pairs with already-
landed substrate (`rge-scene-loader` + golden fixture + runtime-headless
proof of life). The others either require an entirely new widget (#9
hierarchy panel), a host-design decision (#8 menu bar location), or a new
serialization story (#14, #19).

### Why a substrate prerequisite is also real

Row (#11): the `EditorShell` constructor shapes today are `new()`,
`with_world_projection_graph(world, projection, graph)`, and
`with_render_mesh(prebuilt)` (per `crates/editor-shell/src/lib.rs:67`
re-export + `crates/editor-shell/src/lifecycle/mod.rs` docstring at lines 24-166).
`load_scene_into_world(&scene) → World` returns a `rge_kernel_ecs::World`
with no associated `CadGraph` / `CadProjection`, so handing it to
`with_world_projection_graph` requires either a synthetic empty CadGraph
adapter (likely incorrect — the simple-scene has no CAD operators) or a new
constructor shape `EditorShell::with_world(world)` that bypasses both the
CAD pipeline and the `--glb` render-mesh path. That is a code-path
substrate change.

The substrate change is genuinely bounded — `with_render_mesh` is its existing
shape twin: a constructor that accepts a pre-built render-side artifact and
deliberately leaves CAD-side state `None` (per
`crates/editor-shell/src/lifecycle/mod.rs:148-166`, dispatch G's design note).
A symmetric `with_world` would be the same idea on the World axis.

### Why the smallest safe follow-up is docs-only

Phase 9 freeze posture (`plans/PLAN.md:119-134`, `plans/IMPLEMENTATION.md:589-605`)
prefers documenting then implementing only when pressure is demonstrated. Every
prior Phase 9 audit in this delegated-main batch (ISSUE-210/212/214/216/217/218)
returned a docs-only or audit-only smallest-safe-follow-up (see EXEC summaries
in this report's `RELATED_FILES`).

The single safe follow-up here is to write a Phase 9 PREFLIGHT section appended
to `plans/BASELINE.md` (matching the shape of the existing "Editor-usability
preflight (Phase 9)" + "Live-inspector wiring preflight (Phase 9)" entries
referenced in `change.md:1626-1628`). The preflight scopes wiring
`rge-editor` to consume `rge_scene_loader::load_scene_into_world` for an
opt-in `--scene <project-path>` (or `--project <path>`) CLI flag,
enumerating:

1. The missing `EditorShell::with_world(world: KernelWorld)` constructor —
   what it borrows from `with_render_mesh` (skip CAD pipeline) and what it
   differs from (no prebuilt `RenderMesh`; the loaded scene may or may not
   carry mesh-bearing entities; for the current simple-scene fixture there
   are none).
2. Render-path implications for non-CAD scenes — the loaded `World` for the
   simple-scene contains only `Camera` + `KeyLight` + `Transform` /
   `Visibility` / `Light`, so the existing render path's CAD/projection-only
   surface (`render_path::EditorShell::render_frame`) needs an explicit
   blank-viewport branch that doesn't break on `cad_world == None`.
3. Inspector / `InspectorHandoff` / `PlayState` interactions — the loaded
   World participates in `WorldSnapshot` (Play/Stop round-trip) trivially;
   the inspector snapshot already reports tick count + play state without
   any CAD-specific assumptions; PIE behavior is unchanged.
4. Test strategy — mirror
   `runtime/runtime-headless/tests/loads_golden_simple_scene.rs:25-55` with
   an `editor/rge-editor/tests/` integration test that spawns the binary
   with `--scene path/to/.rge-project` and asserts a deterministic stdout
   line + tick advance; the editor doesn't need a winit event loop for the
   load assertion (per the post-v0 `render_frame_e2e_perf.rs` non-winit
   precedent referenced in `Status.md:7`).
5. Forward-bounded scope — explicitly defers ISSUE-212's
   `BusEntry::CadCheckpoint` ADR (row #12), MenuRegistry wiring (row #8),
   hierarchy panel (row #9), Save .rge-scene (row #14), and workspace
   selection (row #19). Those remain documented future-work entries.

This is the smallest action that:
- Decreases the live-tree gap between substrate ready (scene-loader,
  runtime-headless, golden fixture) and user-visible capability (editor can
  open the same fixture).
- Doesn't conflict with ISSUE-212's prior follow-up (the CadCheckpoint ADR
  is orthogonal: simple-scene has no CAD operator, so the BusEntry shape is
  irrelevant for the `--scene` path).
- Doesn't require new tests, new Cargo edges, or any tracked source edit
  in the same dispatch.
- Respects the freeze policy: documents the bounded next implementation
  scope before any code lands.

### Why not other verdicts

- `user_visible_gap_triggered`: would imply naming the implementation
  task itself (the `--scene` wire-up) as the smallest safe follow-up.
  That follow-up would require source/test/Cargo edits and therefore
  `FOLLOW_UP_BRANCH_MODE = PublishMode branch only unless later
  authorized by a human` — defensible, but skips the substrate-design
  step the constructor question forces. The preflight first is more
  conservative.
- `pressure_deferred`: would imply the editor-usability axis has no
  concrete bounded follow-up worth naming. Evidence in rows (#8) - (#10)
  contradicts that; (#10) in particular has substrate ready and a real
  user with a real artifact (the golden fixture) who cannot open it from
  the editor today.
- `needs_human_arbitration`: would imply an unresolved product/design
  decision (e.g., should the editor open `.rge-project` or `.rge-scene`?
  Should the menu bar live in editor-shell or rge-editor? Should the
  Action trait grow?). Several of those exist (rows #8, #14, #16, and
  #12 — the last named historically by ISSUE-212). But for the chosen
  row #10 + #11 pair the path is bounded; #12 is not on the critical
  path for `--scene` load; the preflight-first approach defers the
  open-vs-design questions to the preflight's own enumeration.
