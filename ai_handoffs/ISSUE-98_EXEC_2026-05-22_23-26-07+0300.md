It does **not** re-export `Cache`, `MemoryCache`, or `AssetId` from the stub. So although the module is `pub`, it is not surfaced into the crate's prelude-style root API.
- Workspace search (`rg -n "io_image::asset_store_stub|io_image::Cache|io_image::MemoryCache"`) finds **zero in-tree consumers** of the stub — every reference is in either prior handoff packets or `.ai/dispatch.tasks.md`. The stub's only callers are its own unit tests at `crates/io-image/src/asset_store_stub.rs:76-93`.
- The stub does not depend on `rge-asset-store`, and `crates/io-image/Cargo.toml` confirms `rge-asset-store` is **not** a declared dependency of `rge-io-image` (the manifest's `[dependencies]` block at `crates/io-image/Cargo.toml:12-15` lists only `image`, `exr`, and `thiserror`). The stub is therefore a self-contained island.
- The actual `rge-io-image` public surface — the `Image` data type, codec entry points (`load_path`, `load_bytes`, `load_png`, `load_jpeg`, `load_exr`, `load_hdr`, plus their `save_*` counterparts), and `mip_chain::generate_mip_chain` — is fully reachable and well-exercised by the in-crate codec tests (`crates/io-image/tests/round_trip_*.rs`, `mip_chain_test.rs`), but **none** of that surface participates in or even references the local stub.

**Reachability verdict:** the io-image cache code path exists as a public module but is dead-ended in practice — no in-tree consumer reaches it, and nothing it offers is wired into the codec / `Image` data path that the rest of the workspace actually uses.

### Q2. Cache Surface

The two surfaces are different shapes:

`rge_io_image::asset_store_stub::Cache` (`crates/io-image/src/asset_store_stub.rs:23-30`):

```rust
pub type AssetId = Vec<u8>;
pub trait Cache: Send + Sync {
    fn put(&self, blob: Vec<u8>) -> AssetId;       // &self (interior mutability), infallible
    fn get(&self, id: &AssetId) -> Option<Vec<u8>>;
    fn has(&self, id: &AssetId) -> bool;
}
```

- Reference impl `MemoryCache` (`asset_store_stub.rs:35-74`) uses `Arc<Mutex<HashMap<AssetId, Vec<u8>>>>` plus a counter; `put` mints an id from `counter.to_le_bytes() ++ (blob.len() as u64).to_le_bytes()` (`asset_store_stub.rs:49-58`). The id is **not** derived from the blob's content — two byte-identical `put`s yield two different ids and two stored entries.
- Send + Sync, infallible signatures, no eviction, no `total_size`, no `len`, no `is_empty`. There is no failure type.

`rge_asset_store::Cache` (`crates/asset-store/src/cache.rs:41-87`, re-exported at `crates/asset-store/src/lib.rs:51`):

```rust
pub trait Cache {
    fn get(&self, id: &AssetId) -> Result<Option<Bytes>, CacheError>;
    fn put(&mut self, bytes: Bytes) -> Result<AssetId, CacheError>;        // &mut self, fallible
    fn evict_lru(&mut self, max_bytes: u64) -> Result<(), CacheError>;
    fn total_size(&self) -> u64;
    fn len(&self) -> usize;
    fn is_empty(&self) -> bool { self.len() == 0 }
}
```

- `AssetId` here is re-exported from `rge-kernel-asset` (`crates/asset-store/src/asset_id.rs:10`) and is BLAKE3-keyed via `AssetId::from_bytes(&bytes)` (see usage at `crates/asset-store/src/cache.rs:138` in the `InMemoryCache::put` impl). Contract guarantees include idempotent content-addressed `put`, byte-for-byte round-trip, cross-machine deterministic id, and an eviction-respects-cap LRU policy (`crates/asset-store/src/cache.rs:27-40`, contract bullets).
- `Cache` is **not** `Send + Sync`, takes `&mut self` on `put` / `evict_lru`, is fallible with `CacheError` (`crates/asset-store/src/lib.rs:64-76`), and exposes size / count / eviction methods.

**Surface comparison:** the io-image local stub is a *legacy byte-bag*: `&self` + infallible + counter-keyed + `Send + Sync`. `rge_asset_store::Cache` is a *content-addressed mutable LRU*: `&mut self` + fallible + BLAKE3-keyed + with cap eviction. They are not drop-in compatible at either the trait signature, the id semantics, the failure model, or the synchronization model. The io-image stub doc-comment (`asset_store_stub.rs:1-11`) admits this — it explicitly frames the local trait as a placeholder predating the real crate and asks callers to swap to `rge_asset_store::Cache` once W16 lands. W16 (the asset-store crate) is now present, but the io-image stub still has its pre-W16 shape and has **not** been migrated, nor wired to consume `rge_asset_store::Cache`.

### Q3. Content-Addressed Identity

`rge-io-image` **does not currently expose or derive** a stable content-addressed identity that is compatible with `rge_asset_store::Cache`.

Evidence:

- The `Image` data type (`crates/io-image/src/image_data.rs:14-24`) carries `width: u32`, `height: u32`, `pixel_format: PixelFormat`, and `pixels: Vec<u8>`. It has **no** `content_hash()` method, no `AssetId` accessor, no BLAKE3 derivation, and no canonical-bytes function. Grep for `content_hash|blake3|hash` inside `crates/io-image` returns only the two doc-comment mentions in `asset_store_stub.rs:18` and `:33` — there is zero code that hashes pixel bytes.
- The local `asset_store_stub::AssetId = Vec<u8>` (`asset_store_stub.rs:20`) is explicitly opaque. The reference `MemoryCache::put` synthesises ids from a counter plus blob length (`asset_store_stub.rs:49-58`) — these ids are **non-deterministic across runs**, **non-content-addressed**, and would actively violate the `rge_asset_store::Cache` idempotence + cross-machine determinism contract (`crates/asset-store/src/cache.rs:29-37`).
- io-image declares **no** dependency on `rge-asset-store` (`crates/io-image/Cargo.toml:12-15`), so it does not even have access to `AssetId::from_bytes` from inside its own crate today.
- For comparison, io-gltf carries a per-asset `content_hash()` shape: e.g. `ImageAsset::content_hash()` (referenced from `crates/io-gltf/src/asset_store_cache.rs:217-218` and the `image_canonical_bytes` helper at `crates/io-gltf/src/asset_store_cache.rs:401-413`) returns a typed `ImageHandle`, and the io-gltf adapter chooses the canonical byte form precisely so `AssetId::from_bytes(image_canonical_bytes(&asset)) == AssetId::from_raw(asset.content_hash().0)`. **io-image has no equivalent typed handle or canonical-bytes function.**

A content-addressed identity for an io-image `Image` would therefore have to be **designed**, not extracted: choose a canonicalisation (header order, endianness, pixel-format tag, optional alpha-premultiplication normalisation, palette/colour-space tag), commit to that as the BLAKE3 input, and either expose it as a new `Image::content_hash()` method or as a free function that walks an `&Image`. None of that exists today.
