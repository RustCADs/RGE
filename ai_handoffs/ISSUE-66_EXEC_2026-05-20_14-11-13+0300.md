## Open Questions for Reviewer

- Note for the Reviewer / control role: `git status` still reports
  `crates/editor-ui/tests/phase8_graph_diff_smoke.rs` as `M` after the
  normalization, but `git diff` on that path produces zero content lines and
  `git ls-files --eol` reports `i/lf  w/lf` (index and working tree both LF).
  This is the well-known `core.autocrlf=true` stat-only artifact — Git's
  working-tree filter believes the working copy "should" be CRLF for a tracked
  source file on Windows, so it flags the path as modified even though the
  byte-for-byte content matches the index. Subsequent Git operations on the
  file will silently re-CRLF it (the `LF will be replaced by CRLF the next
  time Git touches it` warning that Git emits). Because `cargo fmt` reads
  the on-disk bytes (which are now LF) and not the post-filter content,
  the format gate passes regardless. The open question for Review is whether
  the dispatch should also adjust `.gitattributes` to pin
  `*.rs text eol=lf` so this state is durable across re-checkouts on Windows;
  I did not change `.gitattributes` because the Correction Packet limited the
  change to "newline style in the one named file" and `.gitattributes` was
  outside the listed scope.
