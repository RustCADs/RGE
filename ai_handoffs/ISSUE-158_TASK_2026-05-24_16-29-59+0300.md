## 5-Question Typed Component Payload Preflight Answer Block

### Q1. Current simple-scene contents and missing facts

Answer: `golden-projects/simple-scene/.rge-project` is a current-schema project
named `simple-scene`, describes "Basic load + transform + camera + light
render", targets `desktop`, has no plugins, and references exactly
`scenes/main.rge-scene` (`golden-projects/simple-scene/.rge-project:1-7`).
The README repeats that target at
`golden-projects/simple-scene/README.md:1-5`.

The referenced scene contains one entity only: id
`0000000000000G000000000000`, name `Camera`, empty `components`, empty
`relations`, and one matching root entity
(`golden-projects/simple-scene/scenes/main.rge-scene:1-15`). The schema test
currently asserts that the root entity is named `Camera`, that
`root.components.is_empty()`, and that `root.relations.is_empty()`
(`crates/rge-data/tests/golden_simple_scene_schema.rs:110-115`). The load+tick
test intentionally copies only entity IDs into a test-local ECS world and does
not inspect or translate components, relations, assets, cameras, lights,
transforms, renderer resources, editor state, scripts, or typed components
(`crates/rge-data/tests/golden_simple_scene_load_tick.rs:11-14`,
`crates/rge-data/tests/golden_simple_scene_load_tick.rs:50-56`).

Missing fixture facts: there is no `Transform` payload, no `Camera` payload, no
`Light` payload, no `Visibility` payload, no light entity, and no schema-only
test that pins any raw `ComponentValue` payload string.

### Q2. Exact RON payload strings

Answer: Current code justifies these candidate raw `ComponentValue.data` RON
strings:

- `Transform`: `(translation:(0.0,0.0,5.0),rotation:(0.0,0.0,0.0,1.0),scale:(1.0,1.0,1.0))`
- `Camera`: `(projection:Perspective(fov_y_radians:1.0471976,near:0.05,far:1000.0),viewport:(0.0,0.0,1.0,1.0),priority:0,is_active:true)`
- `Light`: `(color:(1.0,1.0,1.0),kind:Directional(illuminance_lux:100000.0),affects_indirect:true)`
- `Visibility`: `Visible`

Evidence: `Transform` derives `Serialize`/`Deserialize`, has fields
`translation`, `rotation`, and `scale`, and defines identity values
(`crates/components-spatial/src/transform.rs:22-39`). Existing rge-data scene
fixtures already use the exact no-struct-name `Transform` payload string shape
for component data (`crates/rge-data/tests/fixtures/sample_scene.rge-scene:9-12`,
`crates/rge-data/examples/bake_fixtures.rs:73-77`).

`Camera` derives `Serialize`/`Deserialize`, uses a default `Projection::Perspective`
with `fov_y_radians: std::f32::consts::FRAC_PI_3`, `near: 0.05`, `far: 1000.0`,
and defaults viewport `[0.0, 0.0, 1.0, 1.0]`, priority `0`, and active `true`
(`crates/components-render/src/camera.rs:8-18`,
`crates/components-render/src/camera.rs:31-39`,
`crates/components-render/src/camera.rs:43-66`).

`Light` derives `Serialize`/`Deserialize`, defaults to a white directional
light with `illuminance_lux: 100_000.0` and `affects_indirect: true`
(`crates/components-render/src/light.rs:18-23`,
`crates/components-render/src/light.rs:46-73`). `Visibility` derives
`Serialize`/`Deserialize`, defines `Visible`, `Hidden`, and default
`Inherited`, and has tests round-tripping variants through RON
(`crates/components-visibility/src/visibility.rs:19-28`,
`crates/components-visibility/src/visibility.rs:34-60`).

### Q3. ComponentValue.type_id strings

Answer: The current repo proves the canonical `Transform` string
`rge::components::Transform`, but it does not directly prove canonical strings
for `Camera`, `Light`, or `Visibility`.

Proven string:

- `Transform`: `rge::components::Transform`

Candidate strings by the existing `rge::components::TypeName` convention:

- `Camera`: `rge::components::Camera`
- `Light`: `rge::components::Light`
- `Visibility`: `rge::components::Visibility`

Evidence and risk: `ComponentValue` stores `type_id: String` plus `data:
String`, and the comment says `type_id` names the component type, with
`rge::components::Transform` as the example
(`crates/rge-data/src/scene.rs:83-95`). Existing scene and prefab builders,
fixtures, and comments repeatedly use `rge::components::Transform`
(`crates/rge-data/src/scene.rs:132-135`,
`crates/rge-data/tests/round_trip.rs:75-90`,
`crates/rge-data/tests/fixtures/sample_scene.rge-scene:9-22`,
`crates/rge-data/src/prefab.rs:76-80`). The component crates export the Rust
types as `Transform`, `Camera`, `Light`, and `Visibility`
(`crates/components-spatial/src/lib.rs:32-36`,
`crates/components-render/src/lib.rs:31-40`,
`crates/components-visibility/src/lib.rs:25-28`).

The blocking ambiguity is that `kernel/types::TypeId` is a 128-bit hash of a
fully-qualified Rust path string (`kernel/types/src/type_id.rs:48-64`), while
`Reflect` separately exposes both `TYPE_ID` and `FQ_TYPE_NAME`
(`kernel/types/src/reflect.rs:68-84`). The derive macro's current code builds
`FQ_TYPE_NAME` from `module_path!()` and the Rust type name, while the emitted
`TYPE_ID` const is still `[0u8; 16]` for Phase 1.1
(`crates/macros-reflect/src/codegen.rs:96-113`). The same macro rejects enums
for now (`crates/macros-reflect/src/derive.rs:35-43`), and the component
types inspected here derive `Serialize`/`Deserialize` but not `Reflect`. Because
of that, `rge::components::Camera`, `rge::components::Light`, and
`rge::components::Visibility` are plausible candidates, not proven canonical
strings.

### Q4. Raw payload fixture safety and schema-only tests

Answer: Raw `ComponentValue` payloads are schema-only safe in the narrow sense:
`rge-data` parses `ComponentValue.data` as an inert `String`, and the scene
module documents a two-phase model where callers parse the scene first and
later bridge `ComponentValue::data` through reflection keyed by
`ComponentValue::type_id` (`crates/rge-data/src/scene.rs:17-24`,
`crates/rge-data/src/scene.rs:83-95`). The current load+tick harness already
ignores component payloads (`crates/rge-data/tests/golden_simple_scene_load_tick.rs:11-14`,
`crates/rge-data/tests/golden_simple_scene_load_tick.rs:50-56`).

Schema-only tests that would be safe after human confirmation of the type-id
strings are: update `golden_simple_scene_schema.rs` to assert the expected
`ComponentValue` count, `type_id`, and raw `data` strings on the simple-scene
entities; parse each raw `data` string as `ron::Value` to catch malformed RON
without adding component-crate dependencies; and keep the load+tick test
asserting entity-count and tick behavior without translating component data.
Those tests would not require typed component bridging, renderer/GPU behavior,
asset loading, cook output, screenshot baselines, editor UI, or production
loader changes. They would still require edits to tests and golden-project
files, so they are outside this read-only dispatch.

`rge::components::TypeName` strings or a kernel/types-driven Rust
fully-qualified path/hash convention. Current code proves the `Transform`
string but not the other three canonical strings.

Recommendation: NEEDS_HUMAN
