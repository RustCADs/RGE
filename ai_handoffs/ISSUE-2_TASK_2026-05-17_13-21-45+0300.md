## Acceptance Criteria

- The exact formal soak target is run, not a bounded substitute:
  `cargo test -p rge-script-bench --release --lib script_host::tests::phase_3_memory_soak_one_hour -- --ignored --nocapture`
- The command exits 0 and the cargo test result reports the ignored one-hour test as passed.
- The output includes a `phase3_memory_soak:` line with at least `entities`, `cycles`, `elapsed_s`, `restored_components`, `expected_restored_components`, and `final_counter_sum`.
- The output includes a `phase3_memory_soak_memory:` line with numeric `samples`, `start_rss_bytes`, `end_rss_bytes`, `peak_rss_bytes`, `start_vss_bytes`, `end_vss_bytes`, and `vss_delta_bytes`.
- `elapsed_s` is at least 3600 seconds, `cycles > 0`, and `restored_components == expected_restored_components`.
- `peak_rss_bytes` is numeric and greater than zero on the Windows recorder host.
- `vss_delta_bytes` is captured exactly as printed, including a negative value if the process virtual footprint shrank.
- `crates/script-bench/BASELINE.md` is updated only when all pass criteria above hold. The update must preserve the 2026-05-12 historical row and add a new dated run entry or subsection for this metrics-enabled one-hour run.
- The baseline-doc update states the platform mapping honestly: on Windows, resident memory is process working set and virtual memory is process commit charge, so `vss_delta_bytes` is a commit-charge delta.
- No source code, manifest, lockfile, methodology doc, roadmap doc, status log, protocol file, or unrelated packet changes.
