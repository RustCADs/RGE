## What I Changed

### Source

- `kernel/ecs/src/archetype.rs`: replaced the dense `Column { rows: Vec<ColumnRow> }`
  with sparse `Column { rows: Vec<Option<ColumnRow>> }`. Insertion paths now go
  through a single `Column::set(row, value)` helper that pads with `None`
  placeholders if the target row is past the column's tail, overwrites the
  value at `row` when the slot is already populated (preserving the existing
  per-row change tick to keep `replace`/in-place set semantics), and writes a
  fresh `ColumnRow { tick: 0 }` when the slot transitions from `None` to
  `Some`. Removal (`Column::take`) leaves a `None` placeholder so that other
  rows keep their entity alignment, instead of swap-removing within the
  column. `Archetype::swap_remove_entity` now drives column cleanup off the
  *entity* swap-remove: when removing the last row it truncates each column to
  the new length; otherwise, for every column it either calls
  `Vec::swap_remove(row)` (source row materialised in this column) or simply
  clears `rows[row]` to `None` (source row's entity had no value in this
  column, so the destination must report absence). Added
  `Archetype::has_component_at_row(id, row)` to expose row-specific membership.
  `Archetype::get`, `get_erased`, `iter_component`, and the change-tick
  accessors now all skip absent cells transparently. No `unsafe`, no public
  API removals, no new public API beyond the additive
  `has_component_at_row` (internal: used by the row-specific `EntityRef::contains`).
- `kernel/ecs/src/entity.rs`: `EntityRef::contains<C>()` now calls
  `archetype.has_component_at_row(ComponentId::of::<C>(), self.row)` so it
  agrees with `EntityRef::get::<C>().is_some()` on the same row rather than
  returning a coarse archetype-level "any entity has this column" answer.

### Tests

- `kernel/ecs/src/archetype.rs::tests::first_insert_at_nonzero_row_skips_earlier_rows`:
  pins that inserting `Hp` first on row 1 leaves row 0 absent and row 1
  populated (the original archetype layout asserted `row == col.len()` on
  first insertion, which panicked here).
- `kernel/ecs/src/archetype.rs::tests::remove_keeps_other_rows_aligned`:
  removing the middle row's value does not shift the trailing entity's value
  into the removed row.
- `kernel/ecs/src/archetype.rs::tests::sparse_swap_remove_preserves_alignment`:
  three entities with disjoint column sets; despawning the middle one moves
  the last entity into row 1 and that entity keeps both its component values.
- `kernel/ecs/tests/sparse_rows.rs::heterogeneous_typed_insert_get_contains_query`:
  three entities with disjoint sets of `Health`/`Name`; `get`, `contains`, and
  `query` all agree on each row.
- `kernel/ecs/tests/sparse_rows.rs::contains_agrees_with_get_after_remove`:
  enforces the `EntityRef::contains == EntityRef::get(...).is_some()`
  invariant after a sparse remove.
- `kernel/ecs/tests/sparse_rows.rs::typed_first_insert_at_nonzero_row`,
  `entity_mut_first_insert_at_nonzero_row`, and
  `spawn_with_after_nonzero_row_insert_remains_aligned`: cover the typed
  `World::insert`, `EntityMut::insert`, and `World::spawn_with` paths when
  the first value of a component type lands on row >0 or when later
  `spawn_with` interleaves with a sparse column.
- `kernel/ecs/tests/sparse_rows.rs::sparse_remove_does_not_shift_other_rows` and
  `sparse_replace_inserts_when_absent_and_returns_old_when_present`: pin
  remove/replace semantics for sparse columns.
- `kernel/ecs/tests/sparse_rows.rs::sparse_despawn_preserves_swapped_entity_components`,
  `sparse_despawn_when_swapped_entity_lacks_a_column`, and
  `sparse_despawn_of_last_row_truncates_columns`: pin `World::despawn`
  swap-remove correctness across all three sparse cases (swapped entity
  populated, swapped entity absent in the column, and trailing-row removal).
- `kernel/ecs/tests/sparse_rows.rs::heterogeneous_snapshot_round_trip_exercises_erased_nonzero_row_insert`:
  three entities (a < b < c by ULID) with disjoint `Health`/`Name`/`Speed`
  presence; the restore order is fixed by ULID sort so `Speed`'s first ever
  erased insert during restore lands on row 2. The test asserts every row's
  presence/absence is reproduced, that `contains` agrees with `get`, and that
  re-serializing after restore yields byte-identical bytes (snapshot wire
  format unchanged).
