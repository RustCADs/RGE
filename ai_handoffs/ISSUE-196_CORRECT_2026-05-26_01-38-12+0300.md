## Approved Corrections (Planner Sign-Off)

The Executor MUST act on exactly these corrections, nothing more and nothing
less.

1. **Do not select new work after ambiguous recovery visibility** - Codex
   control finding: after a successful recovery mutation,
   `Invoke-AiDispatchAuto.ps1` currently only warns if recovered issue
   visibility under `ai-dispatch` is not confirmed after 60 seconds, then can
   continue into the normal empty-queue path and select/file a new autonomous
   task in the same tick. Required change: after the recovery mutation
   succeeds, if the recovered issue is not confirmed listable for queue drain,
   skip or exit the current autonomous tick before any cap check, Codex task
   selection, `gh issue create`, or queue invocation can run. Acceptance:
   a mocked or non-mutating focused check demonstrates the post-mutation
   visibility-miss path exits 0 or otherwise stops the tick, and does not reach
   `Queue is empty; asking Codex to select the next task...`.

2. **Make visible recovery drain deterministic** - Codex control finding: the
   post-recovery path can still depend on later label-index reads for the same
   recovered issue. Required change: when the recovered issue is confirmed
   visible after mutation, force that issue into the queue-drain path or seed
   the queue state from the confirmed recovery result so stale follow-up label
   search results cannot route the tick into new task selection. Acceptance:
   a mocked or non-mutating focused check demonstrates the post-mutation
   visibility-confirmed path drains the recovered issue, or invokes the normal
   queue drain with that issue pending, and never selects/files a new task in
   the same tick.
