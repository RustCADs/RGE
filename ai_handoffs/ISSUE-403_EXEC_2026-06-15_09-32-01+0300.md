## Read-Only Audit Answers

### Q1: Current CommandBus public contract and Action context

`rge-editor-actions` documents `CommandBus::submit` as the single mediation
layer for editor mutations into runtime state at `crates/editor-actions/src/lib.rs:10`.
The public `Action` trait is `Send + Sync + 'static` and currently exposes
`name`, `id`, `apply(&self, world: &mut World)`, `revert(&self, world: &mut World)`,
optional `merge`, and `payload` in `crates/editor-actions/src/action.rs:74-112`.

`CommandBus::submit` takes `Box<dyn Action>` plus `&mut rge_kernel_ecs::World`,
then handles coalescing, `Action::apply`, audit-ledger projection, undo-stack
push, and cursor advancement in `crates/editor-actions/src/bus.rs:143-199`.
`undo`, `redo`, `mark_saved`, and `is_dirty` are owned by
`crates/editor-actions/src/bus.rs:216`, `:245`, `:277`, and `:284`, with the
underlying `UndoStack` save mark/dirty comparison in
`crates/editor-actions/src/undo_stack.rs:70-85`.

The shell wrapper delegates bus calls to `self.world.kernel_mut()` in
`crates/editor-shell/src/lifecycle/commands.rs:21-25` and exposes
`submit_action`, `undo_command`, `redo_command`, `mark_saved_command`, and
`command_bus` at `crates/editor-shell/src/lifecycle/commands.rs:274-314`.
The only production editor-shell `Action` implementation found is
`SetTimeScale` in `crates/editor-shell/src/lifecycle/commands.rs:188-239`;
its submit path is `set_time_scale` -> `submit_action` at
`crates/editor-shell/src/lifecycle/commands.rs:356-390`.

Conclusion: the bus is real and authoritative for the world it receives, but
the current action context is World-only. It cannot directly mutate the
separate CAD graph/projection state owned by `EditorShell`.

### Q2: Current authority for CAD projection, selected-CAD bounds, open/save/load, undo, dirty state, and save marks

CAD runtime state is owned in `EditorShell`, not inside the bus context:
`cad_world`, `projection`, `cad_graph`, and `cad_entity` live at
`crates/editor-shell/src/lifecycle/mod.rs:354-375`. The default cuboid demo
constructs a `CadGraph`, commits a cuboid, spawns one projected entity, ticks
projection, then passes the triple into `EditorShell::with_world_projection_graph`
in `editor/rge-editor/src/main.rs:867-904`; the shell constructor stores those
fields in `crates/editor-shell/src/lifecycle/mod.rs:1080-1122`.

`CadGraph` itself is transactional: mutations are bracketed by
`begin_operation`/`commit`, and `graph_mut` is only valid while an operation is
open in `crates/cad-core/src/checkpoints/mod.rs:194-279`. The underlying
`OperatorGraph` exposes additive graph construction through `add_operator`,
`connect`, and `set_root` in `crates/cad-core/src/graph/operator_graph.rs:151-204`.
Operators include `Cuboid`, `Transform`, and the other CAD variants in
`crates/cad-core/src/operators/mod.rs:195-213`.

Projection authority is `CadProjection`: `tick` synchronizes dirty `BRepHandle`
entities with the current `CadGraph` head per `crates/cad-projection/src/lib.rs:38-52`
and is implemented at `crates/cad-projection/src/lib.rs:502-554`.
`spawn_brep_entity` and `despawn_brep_entity` manage entity/projection bindings
at `crates/cad-projection/src/lib.rs:454-483`. Render conversion is read-only
through `CadProjection::render_mesh_for` at
`crates/cad-projection/src/render_adapter.rs:90-106`, and render upload consumes
that mesh from shell state at `crates/editor-shell/src/render_path.rs:990-1001`.

Selected-CAD bounds are computed by shell code, not projection code:
`selected_cad_scene_bounds` reads `coord.selection`, `projection`, and
`cad_world`, then maps selected entities through `render_mesh_for` at
`crates/editor-shell/src/lifecycle/mod.rs:2084-2100`; the double-click frame path
uses it at `crates/editor-shell/src/lifecycle/mod.rs:2122-2128`.

Open/load authority is in `handle_open_request`: it gates on Editing, gets a
path from the injected dialog, loads scene/project worlds through the injected
hook, and calls `replace_world` only after successful load in
`crates/editor-shell/src/lifecycle/open_request.rs:233-341`. `replace_world`
clears CAD fields, render state, selection, command bus, source pointers, and
clipboard at `crates/editor-shell/src/lifecycle/mod.rs:910-944`.

Save authority is in `handle_save_request` and Save-As code:
`crates/editor-shell/src/lifecycle/save_request.rs:205-270` writes by current
`SaveSource`, commits a new source on Save-As, and calls `mark_saved_command`
only after successful writes. New-project Save-As follows the same pattern at
`crates/editor-shell/src/lifecycle/save_request.rs:355-397`.

Undo, redo, dirty state, and save marks remain CommandBus authority:
menu commands route `Undo`/`Redo` to `undo_command`/`redo_command` in
`crates/editor-shell/src/render_path.rs:427-442`; close-family dirty guards call
`command_bus().is_dirty()` at `crates/editor-shell/src/lifecycle/mod.rs:1014`.

### Q3: Candidate first mutations possible, and which require wider decisions

- Adding a CAD primitive is possible as a future implementation if a human
  approves the bus context expansion. Existing source already has the required
  CAD graph/projection primitives: `CadGraph::begin_operation`/`commit`,
  `OperatorGraph::add_operator`/`set_root`, and
  `CadProjection::spawn_brep_entity`/`tick`. The missing piece is the
  CommandBus action context: `Action::apply` currently receives only
  `&mut World`, so a CAD add action cannot update `cad_graph`, `cad_world`,
  `projection`, and render refresh state without an approved adapter/context.
- Deleting selected CAD geometry is not the smallest first mutation. Although
  `CadProjection::despawn_brep_entity` exists, deletion also needs graph/root
  deletion semantics, selected entity/node mapping, face-selection pruning, and
  undo-safe projection/render invalidation. The targeted inverse search found
  no public `remove_operator`/`delete_operator` API in CAD core.
- Transforming selected CAD geometry is possible at the CAD-core operator
  level (`TransformOp` exists), but as an editor command it requires selection
  ownership, transform product UX, a graph wiring policy, and the same
  bus-context decision before it can be coherent.
- Editing primitive parameters is wider than adding one primitive. It needs an
  editable inspector/property model, parameter identity/rebuild policy, and
  face/selection preservation decisions in addition to the bus-context bridge.
- Clearing or replacing the CAD graph is technically close to existing
  lifecycle reset paths, but it primarily exercises document/reset authority
  rather than a real CAD authoring mutation and would overlap open/new/save
  semantics.
- Treating open/new/save transitions as the first mutation boundary is already
  lifecycle-owned through `replace_world`, `handle_open_request`, and
  `handle_save_request`; it does not answer how a user-authored CAD operation
  flows through CommandBus, undo/redo, dirty, and projection.

### Q4: Exact human decision needed before implementation

Approve or reject a narrow CommandBus context expansion for editor-owned CAD
mutations. The concrete approval should say whether a future implementation may
change the `Action`/`CommandBus::submit` contract from World-only to an
editor-command context that includes the kernel world plus shell-owned CAD
state needed for a reversible CAD add action: `cad_world`, `cad_graph`,
`CadProjection`, selected/render identity, and render refresh hooks.

If approved, the first implementation should be limited to one bus-routed
"add CAD cuboid primitive" action as proof of the boundary. If rejected, a
future planner should not implement CAD mutation through a parallel non-bus
channel without a separate architecture decision, because that would fork the
undo/dirty/audit story.

### Q5: Single smallest coherent recommended first mutation

Recommend approving "add one CAD cuboid primitive through CommandBus" as the
first implementation boundary, paired with the minimal editor command context
needed to make it honest. It is the smallest coherent mutation because it
exercises creation, commit, projection, render visibility, undo/revert,
dirty-state movement, save-mark interaction, and save/load authority without
also needing deletion policy, transform UX, parameter-inspector editing, or
document lifecycle replacement semantics.
