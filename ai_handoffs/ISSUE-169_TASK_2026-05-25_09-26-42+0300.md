## Goal

Fix `rge-kernel-ecs` so its existing single catch-all archetype supports sparse
component rows correctly. In the current shape, component columns implicitly
assume dense row alignment: first insertion of a component type at row 0 works,
but first insertion at a later entity row can panic or misalign component rows,
and `EntityRef::contains<C>()` reports global column existence rather than
whether the specific entity row has `C`. This dispatch must make heterogeneous
component sets work inside the current public ECS API without introducing
archetype migration, new registries, snapshot format changes, or cross-crate
scope.

## Deliverables

- Rework the `kernel/ecs` archetype component-column storage so a column can
  represent "component absent for this entity row" while preserving row-to-entity
  alignment for every live entity row in the catch-all archetype.
- Make typed insertion paths, including `World::insert<C>()`,
  `World::replace<C>()`, `EntityMut::insert<C>()`, and `World::spawn_with<C>()`,
  handle first insertion of a component type at any existing entity row without
  panic, row shifting, or misaligned values.
- Make the erased insertion path used by snapshot restore,
  `World::insert_erased`, handle first insertion of a component type at a
  nonzero entity row without panic or row misalignment.
- Make `EntityRef::contains<C>()` row-specific: it must return `true` only when
  that entity row currently has a `C` value, not merely because a `C` column
  exists somewhere in the archetype.
- Preserve existing behavior for `get`, `get_mut`, `query`, `remove`,
  `replace`, `despawn` swap-remove, and snapshot serialize/restore, adjusted
  only so absent sparse cells are skipped rather than treated as components.
- Add focused `kernel/ecs` tests covering heterogeneous typed
  insert/get/contains/query, nonzero-row first insert, sparse remove/replace,
  sparse despawn swap-remove, and heterogeneous snapshot round-trip.

## Acceptance Criteria

- An entity can lack a component value that another entity has, even though both
  live in the same catch-all archetype.
- `World::insert<C>()` and `World::insert_erased` both work when the first ever
  value of component type `C` is inserted at row greater than zero.
- `EntityRef::contains<C>()` agrees with `EntityRef::get<C>().is_some()` for the
  same entity row in heterogeneous worlds.
- `World::query<C>()` yields exactly the entity rows that currently contain `C`
  and does not yield rows with absent sparse cells.
- `World::remove<C>()` removes only the target entity row's component value and
  does not shift other component values out of entity alignment.
- `World::replace<C>()` returns the old value when present, inserts when absent,
  and preserves all unrelated component rows.
- `World::despawn` swap-remove preserves component values and absences for the
  entity moved into the removed row across all sparse columns.
- Snapshot serialization includes only registered component values actually
  present on each entity row, and restore reproduces heterogeneous component
  presence/absence without changing the existing snapshot wire format.
- Tests exist in `kernel/ecs` for all of these scenarios:
  heterogeneous typed insert/get/contains/query; first insertion of a component
  type at a nonzero row through typed insertion; sparse remove and replace;
  sparse despawn swap-remove; heterogeneous snapshot round-trip that also
  exercises erased insertion of a component type first restored at a nonzero
  row.
- The implementation introduces no `unsafe` code and does not add `Reflect`,
  global registries, wrapper component types, new `SnapshotComponent` behavior,
  snapshot wire-format changes, new public ECS APIs, or breaking changes to the
  existing public ECS API.

## Planner Notes

The likely fault line is `kernel/ecs/src/archetype.rs`: columns are currently
dense `Vec` storage, `insert_component` and `insert_erased` assert that the
target row equals the column length, and `remove_component` swap-removes within
one component column. That model is incompatible with heterogeneous rows in a
single catch-all archetype. A valid fix should make column rows explicitly able
to hold absence and should keep column row indexes aligned with
`Archetype::entities()` through insertion, removal, query, and entity
swap-remove.

Do not paper over the issue by making every entity carry default placeholder
components or by wrapping component types. Absence must remain observable as
`get::<C>() == None`, `contains::<C>() == false`, and no query result for that
entity/component pair. Snapshot restore is the important erased-path proof:
include a heterogeneous snapshot test where the restore path first inserts at
least one registered component type on a nonzero row.
