## Goal

Create the narrow `rge-scene-loader` bridge crate requested by GitHub issue
#171. The crate must load an already-parsed `rge_data::Scene` into an
`rge_kernel_ecs::World`, preserving scene ULIDs as ECS entity IDs and inserting
the four current simple-scene component types through the typed ECS
`World::insert` API. This is the retry of the Scene-to-World bridge after the
component crates and sparse ECS rows were prepared by prior work; it must not
re-open schema, ECS, reflection, renderer, editor, runtime, or golden-fixture
scope.

## Deliverables

- Add exactly one new workspace member, `crates/rge-scene-loader`, to the root
  workspace member list.
- Create a new library crate named `rge-scene-loader` under
  `crates/rge-scene-loader` using existing workspace package metadata and
  workspace lints.
- Add only the dependencies needed for this bridge: `rge-data`,
  `rge-kernel-ecs`, `rge-components-spatial`, `rge-components-render`,
  `rge-components-visibility`, `ron`, and a small error dependency such as
  `thiserror` only if the implementation uses it. Use path dependencies for
  internal crates that are not already present in root `[workspace.dependencies]`
  rather than editing root workspace dependency metadata.
- Expose a public Scene-to-World loader API. The required public entry point is
  `pub fn load_scene_into_world(scene: &rge_data::Scene) -> Result<rge_kernel_ecs::World, SceneLoadError>`.
  Additional private helpers are allowed inside the new crate only.
- Preserve every `rge_data::EntityId` ULID by converting with
  `rge_kernel_ecs::EntityId::from_ulid(...)` and creating the ECS entity with
  `World::spawn_with_id(...)`. Do not call `World::spawn()` for scene entities.
- Spawn all scene entities before component insertion so every later
  `World::insert` targets an entity that already exists.
- Implement an explicit match table over `ComponentValue.type_id.as_str()` for
  exactly these four strings and no additional component type IDs:
  `rge::components::Transform`, `rge::components::Camera`,
  `rge::components::Light`, and `rge::components::Visibility`.
- For each supported component arm, deserialize the raw
  `ComponentValue.data` with the matching typed call:
  `ron::from_str::<Transform>`, `ron::from_str::<Camera>`,
  `ron::from_str::<Light>`, or `ron::from_str::<Visibility>`.
- Insert the parsed original component values through the current typed
  `rge_kernel_ecs::World::insert` API. The inserted types must be the component
  crate types themselves, not wrapper types.
- Return a loader error for unsupported `ComponentValue.type_id` values or
  failed typed RON deserialization. Do not silently skip unsupported or invalid
  components.
- Add focused `rge-scene-loader` tests using the tracked
  `golden-projects/simple-scene` fixture. The tests must prove that the loaded
  `World` contains:
  - the golden `Camera` entity with `Transform`, `Camera`, and
    `Visibility::Visible`; and
  - the golden `KeyLight` entity with `Transform` and `Light`.

## Acceptance Criteria

- The root workspace has exactly one new member: `crates/rge-scene-loader`.
- No root workspace dependency metadata or unrelated root manifest content is
  changed.
- The new crate compiles as package `rge-scene-loader` and exposes
  `load_scene_into_world(&rge_data::Scene) -> Result<World, SceneLoadError>`.
- The loader converts every scene entity ID through
  `rge_kernel_ecs::EntityId::from_ulid` and calls `World::spawn_with_id` for
  each scene entity before inserting components.
- The component bridge is a literal four-arm match table for only
  `rge::components::Transform`, `rge::components::Camera`,
  `rge::components::Light`, and `rge::components::Visibility`.
- Each supported component arm uses the typed `ron::from_str::<T>` call for the
  matching component type and then calls `World::insert(entity_id, parsed_value)`.
- The implementation does not use `Reflect`, `kernel/types`, `inventory`,
  `linkme`, any global registry, any registration macro, `SnapshotComponent`,
  wrapper component types, `World::insert_erased`, `Box<dyn Any>`, `TypeId`, or
  any other type-erased component insertion path.
- The golden simple-scene test locates or otherwise pins the `Camera` entity and
  asserts that `world.entity(camera_id).unwrap().get::<Transform>()`,
  `get::<Camera>()`, and `get::<Visibility>()` are present, with the visibility
  value equal to `Visibility::Visible`.
- The golden simple-scene test locates or otherwise pins the `KeyLight` entity
  and asserts that `world.entity(light_id).unwrap().get::<Transform>()` and
  `get::<Light>()` are present.
- The tests must read the golden fixture but must not modify
  `golden-projects/**`.
- `Cargo.lock`, if changed, is limited to the new `rge-scene-loader` package
  entry and its dependency edges to already-existing packages.

## Planner Notes

The expected implementation is intentionally small: parse a `Scene` that the
caller already has, create a fresh `World`, preserve scene IDs through
`EntityId::from_ulid` plus `spawn_with_id`, then walk each component envelope
through an explicit four-string `match` and typed RON deserialization. Unknown
component type IDs should be surfaced as a loader error so future scene content
does not disappear silently.

The existing golden fixture already pins the two relevant entities:
`Camera` has `Transform`, `Camera`, and `Visibility`; `KeyLight` has
`Transform` and `Light`. Tests in the new crate may use the canonical entity ID
strings from `golden-projects/simple-scene/scenes/main.rge-scene` or locate the
scene entities by `name` before converting their ULIDs to ECS IDs.
