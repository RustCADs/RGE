## Goal

Make RGE data fixture and project file line endings deterministic on Windows checkouts so `rge-data` byte-identical round-trip tests and `.ai/dispatch.verify.ps1` do not depend on stale Cargo target-cache state or local `core.autocrlf` behavior.

## Context

ISSUE-233 blocked after scoped physics tests passed because the canonical workspace gate failed in `cargo test -p rge-data --test round_trip`: `include_str!` embedded CRLF fixture bytes from a Windows checkout while the canonical RON serializer emits LF. These files are source data formats with byte-identical tests, so their line endings must be pinned in git attributes and normalized in the checkout.

## Deliverables

1. Add `.gitattributes` rules that pin RGE text data formats to LF on checkout and commit:
   - `*.rge-project text eol=lf`
   - `*.rge-scene text eol=lf`
   - `*.rge-prefab text eol=lf`
2. Ensure the affected RGE fixture/project files in this checkout are physically LF before running tests. Use repo-safe normalization such as `git add --renormalize` plus rewriting or re-checking files as needed; do not alter semantic content.
3. Verify `git ls-files --eol` reports `w/lf` for the tracked `crates/rge-data/tests/fixtures/*.{rge-project,rge-scene,rge-prefab}` files and `golden-projects/simple-scene` RGE files.
4. Do not change Rust source. This is a line-ending policy fix only.
