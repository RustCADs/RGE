## Open Questions for Reviewer

- The underlying cause is the repo-wide `core.autocrlf=true` Git
  configuration on this Windows worktree: it smudges committed LF files to
  CRLF on checkout, after which `cargo +nightly fmt --check` (which
  enforces `newline_style = "Unix"`) trips on the working-tree copy even
  though the indexed copy is already LF. After this rewrite,
  `git status` reports the file as `M` and the diff prints only the
  "LF will be replaced by CRLF the next time Git touches it" advisory —
  this is autocrlf bookkeeping, not a semantic change. A follow-up
  outside the ISSUE-56 scope could add a `.gitattributes` entry
  (e.g. `*.rs text eol=lf`) so future checkouts do not re-introduce
  the same failure mode. Flagging only; explicitly NOT changed in this
  correction.
