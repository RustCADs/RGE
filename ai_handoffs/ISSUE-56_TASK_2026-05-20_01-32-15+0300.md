## Goal

Add a focused unit-test smoke in `rge-material-graph` proving that the existing
graph-foundation invalidation router can propagate dirty state through a
`MaterialGraph` using the graph substrate's outgoing-edge view. This closes
GitHub issue #56 by pinning the material graph wrapper's compatibility with
`Invalidation::mark_dirty` without adding material shader, runtime, editor, or
new public API behavior.

## Deliverables

- One new `#[test]` inside the existing `#[cfg(test)] mod tests` in
  `crates/material-graph/src/lib.rs`.
- The test builds a small `MaterialGraph` of material nodes connected by typed
  material edges.
- The test registers an `InvalidationListener`, calls
  `Invalidation::mark_dirty`, and supplies the dependents closure by reading
  the underlying graph's outgoing edges, e.g. via `g.graph.outgoing(node)` and
  `g.graph.edge(edge_id).map(|record| record.dst)`.
- The test asserts the exact listener call sequence, proving downstream
  material nodes are reached exactly once and in deterministic order.

## Acceptance Criteria

- `crates/material-graph/src/lib.rs` contains exactly one new focused
  invalidation propagation unit test.
- The test uses `rge_kernel_graph_foundation::Invalidation` and
  `InvalidationListener`; it does not reimplement invalidation traversal.
- The test's propagation closure derives dependents from the `MaterialGraph`
  substrate graph's outgoing edges rather than from an ad hoc side table.
- The asserted result includes the dirty root and downstream material nodes in
  a deterministic `Vec<NodeId>` order, with each expected node appearing once.
- `cargo test -p rge-material-graph` exits 0.

## Planner Notes

- `MaterialGraph` stores its substrate graph in a private `graph` field. The
  inline unit tests in the same module can access that field, so no production
  accessor should be needed for this smoke.
- `Graph::outgoing(NodeId)` yields deterministic `EdgeId` iteration from the
  BTree-backed adjacency set, and `Graph::edge(edge_id)` exposes each
  `EdgeRecord` destination. Use that to build the `dependents_of` closure for
  `Invalidation::mark_dirty`.
- Keep the graph small and the assertion direct. A linear material chain is
  enough if the listener log equals the exact expected node sequence; a small
  fanout/diamond shape is also acceptable if the expected order is pinned and
  duplicate downstream delivery is asserted absent.
