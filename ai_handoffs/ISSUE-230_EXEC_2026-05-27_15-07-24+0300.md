## What I Changed

### Source

- `Invoke-AiDispatchQueue.ps1`:
  - Added the pure helper `Resolve-DispatchPrViewMetadata` right after
    `Format-DispatchPrBody`. The helper takes `ExitCode` and `Text` (the raw
    `gh pr view --json number,url` outputs from `Invoke-Tool`) and returns
    `[pscustomobject]@{ Success; PrNumber; PrUrl; FailureReason }`. `Success`
    is `$true` only when (a) the call exited 0, (b) the JSON parses, (c) the
    payload is non-null, and (d) both `number > 0` and `url` is non-empty
    after string coercion. Every other case sets `Success=$false` with a
    `FailureReason` that names the specific failure (non-zero exit,
    empty output, unparseable JSON, null payload, or incomplete metadata),
    leaves `PrNumber=0`, and leaves `PrUrl=''`. No file I/O, no gh/git
    invocation, no network — pure and deterministic.
  - Rewrote the post-`gh pr create` block (was `Invoke-AiDispatchQueue.ps1`
    near lines 1925-1962) so PR-mode success now flows through the new
    helper. After `gh pr create` returns 0 the queue ALWAYS runs
    `gh pr view <branch> --repo <slug> --json number,url`, hands the result
    to `Resolve-DispatchPrViewMetadata`, and:
      - if `Success`, sets `$prNumber`, `$prUrl`, `$published = $true`,
        `$publishedSha = $commitSha`, and `$publishDetail = "Pushed $branch
        to origin and opened PR #$prNumber ($prUrl)."`;
      - otherwise sets `$publishFailed = $true`, `$publishHardFailed = $true`,
        leaves `$published = $false`, leaves the dispatch branch in place for
        human recovery, and sets `$publishDetail` to the helper's
        `FailureReason` plus the explicit note that `gh pr create reported
        success` but PR metadata is incomplete.
    The legacy stdout URL parse of `gh pr create` is removed — the canonical
    source of truth is the `gh pr view --json number,url` payload, which is
    exactly what the correction packet recommended. Existing branch-push and
    `gh pr create` failure handling is unchanged; this correction extends the
    same `publishHardFailed=$true` treatment to missing or incomplete PR
    metadata.
  - Simplified the `'pr'` branch of `$branchLine` (the result-comment line)
    to a single string that emits both PR number and URL. The previous
    fallback branches (URL-only and "PR URL not parsed") are now structurally
    unreachable because the gate above guarantees both fields are present
    whenever `$published = $true`. The acceptance gate from the CORRECT
    packet — "PR-mode success is possible only when the final issue comment
    includes both the PR number and URL" — is now enforced by construction
    rather than by inspection.

### Tests

- `tools/dispatch-tests/Resolve-DispatchPrViewMetadata.Tests.ps1` (new): 17
  cases organised by outcome class.
  - Success path (3 cases): the helper exposes after dot-source, returns
    `Success=$true` with both `PrNumber` and `PrUrl` populated when JSON has
    both fields, returns a `System.Boolean` for `Success`, and is
    deterministic across repeated calls.
  - Non-zero exit (2 cases): `Success=$false` even when the JSON would
    otherwise be valid, `FailureReason` names `gh pr view`, `exit <code>`,
    and the stderr/stdout snippet.
  - Empty / whitespace / null output (3 cases): `Success=$false` with a
    `FailureReason` mentioning `empty output`.
  - Unparseable JSON (1 case): `Success=$false` with a `FailureReason`
    mentioning `unparseable JSON`.
  - Incomplete metadata (5 cases): missing number, missing url, number=0,
    empty-string url, both fields missing — each yields `Success=$false`
    with a `FailureReason` mentioning `incomplete PR metadata`.
  - Failure result shape (2 cases): the result object always carries the
    four documented properties `Success,PrNumber,PrUrl,FailureReason`, and a
    failure `FailureReason` never leaks the success-path "Pushed ... to
    origin ..." sentence.
  All tests dot-source the queue via the existing
  `RGE_AI_DISPATCH_QUEUE_SKIP_MAIN` seam — no `gh`, no `git push`, no
  `gh pr create`, no `gh pr view`, no `Register-ScheduledTask`, no network.
