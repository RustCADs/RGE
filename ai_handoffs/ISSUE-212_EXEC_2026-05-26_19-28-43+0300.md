## Audit findings

### Sharpest current abstraction pain point

**The `editor-actions::Action` trait's `apply(&self, world: &mut World)` and
`revert(&self, world: &mut World)` World-only signatures cannot mediate
CAD-store (CadGraph + CadProjection) mutations, blocking the natural next
class of user-driven editor dispatches.** The PLAN §6.16.3 two-layer Command
Bus model with a `BusEntry::CadCheckpoint { id, description }` variant is
specified but unimplemented; the `BusEntry` enum currently ships only the
`Action(Box<dyn Action>)` variant with `#[non_exhaustive]` marking the future
slot.

Concrete file:line evidence (re-read this dispatch):

- `crates/editor-actions/src/action.rs:87`:
  `fn apply(&self, world: &mut World) -> Result<(), ActionResult>;` — the
  trait method that every editor mutation must implement; binds the Action
  to `rge_kernel_ecs::World` only.
- `crates/editor-actions/src/action.rs:96`:
  `fn revert(&self, world: &mut World) -> Result<(), ActionResult>;` —
  mirror signature for undo; same constraint.
- `crates/editor-actions/src/bus.rs:21–25`: `#[non_exhaustive]` `BusEntry`
  enum currently has exactly one variant, `Action(Box<dyn Action>)`. PLAN
  §6.16.4's planned `CadCheckpoint { id: CadCheckpointId, description:
  String }` variant is reserved by the `#[non_exhaustive]` marking but
  not implemented.
- `crates/editor-actions/src/bus.rs:33–48`: `BusEntry::apply` /
  `BusEntry::revert` dispatch over the (currently single) `Action` variant
  via the `(&mut World)`-only Action trait, so even adding a parallel
  `BusEntry` variant would not give an `impl Action` access to anything
  beyond `World`.
- `crates/editor-actions/src/bus.rs:143`:
  `pub fn submit(&mut self, action: Box<dyn Action>, world: &mut World)
  -> Result<(), BusError>` — the single mutation surface for the Command
  Bus is `(action, &mut World)`.
- `crates/editor-shell/src/lifecycle/mod.rs:492–507`: live comment block
  acknowledging the gap verbatim — "the `Action` trait is `(&mut
  rge_kernel_ecs::World)`-only. CAD-graph and projection mutations are
  intentionally NOT on the bus today — they wait for a future
  'CAD-state into ECS' design dispatch with its own preflight (see
  `plans/BASELINE.md` editor-usability preflight, §F → SpawnCuboidAt
  rejection note)."
- `crates/editor-shell/src/lifecycle/mod.rs:507`: `EditorShell.command_bus:
  CommandBus` field is the bus owned per editor session.
- `crates/editor-shell/src/lifecycle/commands.rs:193–264`: `SetTimeScale`
  is the **first and only production `Action` impl** in the workspace.
  Its `apply` (line 233) and `revert` (line 241) both touch a `World`
  *resource* (`TimeScale`) — exactly the shape the World-only signature
  fits. A CAD-edit action cannot use this shape.
- `crates/editor-shell/src/lifecycle/commands.rs:280`: shell-side adapter
  `submit_action` calls `self.command_bus.submit(action, self.world.kernel_mut())`
  — the bus receives only `&mut KernelWorld`, not the `CadGraph`,
  `CadProjection`, or other editor-shell-owned stores.
- `crates/editor-shell/src/lifecycle/mod.rs:591–593`: editor-shell holds
  `cad_world: Option<CadWorld>`, `projection: Option<CadProjection>`,
  `cad_graph: Option<CadGraph>` as private fields outside the World; the
  Command Bus has no path to them.
- `plans/PLAN.md` §6.16.3 / §6.16.4 (around line 921): the planned
  CAD-graph-checkpoint API
  (`begin_operation` / `commit(op) -> CadCheckpointId` / `rollback` /
