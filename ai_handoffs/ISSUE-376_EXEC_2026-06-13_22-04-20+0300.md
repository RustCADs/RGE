## Candidate Comparison

- Camera/navigation: wheel zoom, right-button orbit, middle-button pan, and
  left-double-click frame-all are all shipped inside `editor-shell` and share the
  existing viewport hit-test and `EditorCameraState` boundary. A further useful
  camera task now implies frame-selected/world-AABB selection, pointer
  capture/window-grab, camera persistence/state policy, or a broader camera
  controller. That is no longer a one-dispatch local gesture.

- Host-shell FIFO replacement/generalized execution: current source still routes
  menu and palette activations through `MenuCommandHandoff` into
  `EditorShell::route_menu_command`. Replacing that route is architecture work,
  not a small implementation follow-up.

- Real plugin command execution: `Command::Custom` / `Command::Plugin` entries
  can be captured and handed to an injected handler, but real execution requires
  runtime/discovery/loading/capability policy. That crosses the task packet's
  plugin-runtime stop condition.

- Keybinding/remap/conflict policy: `AcceleratorTable` and conflict rows exist,
  and the host exposes diagnostics, but remapping, fatal conflicts, or user
  policy are not decided in current source.

- OS/typed clipboard: source has shell-local entity clipboard behavior for
  Cut/Copy/Paste, but no OS clipboard or typed component semantics. Selecting
  this would require product semantics for external clipboard formats and CAD
  identity.

- Authoritative CAD/editor mutation through `CommandBus`: `CommandBus::submit`,
  undo/redo, save mark, and dirty state exist, but `Action::apply` and
  `Action::revert` still take only `&mut World`. CAD/projection mutation through
  the bus needs a richer authority/undo/dirty model before it is a bounded code
  task.
