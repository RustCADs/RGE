## What I Changed

### Source

- `Invoke-AiDispatchQueue.ps1` (`Get-QueueStatusEntries`, ~line 462–550):
  - Extended the porcelain short-status alphabet from
    `[ MADRCU?!]` to `[ MADRCUT?!]`. `T` is git's documented type-change
    code (e.g., regular file -> symlink). With the prior alphabet a
    `T  path` or ` T path` record was silently dropped before the scope
    guard's path validation and would then have been swept into the broad
    `git add -A` that follows.
  - Split the prior single-pass regex check into two phases so the parser
    can distinguish "not a porcelain record" from "porcelain-shaped but
    unrecognized status code":
    1. Column 3 must be a space; if not, the line is not a porcelain
       short-status record and is skipped (defense in depth — stderr is
       already separated, so this only matters if some future git release
       emits a non-record info line to stdout).
    2. Columns 1-2 must match `[ MADRCUT?!][ MADRCUT?!]`. Anything else
       calls `Fail` with the offending line attached. This makes a future
       git status-alphabet expansion fail closed before staging instead of
       silently bypassing the guard.
  - Function header comment updated to document the new alphabet (`T`
    included) and the new fail-closed branch on unknown porcelain codes.

## Open Questions for Reviewer

- The fail-closed branch on unknown porcelain codes includes the offending
  line verbatim in the `Fail` message. That is helpful for diagnostics but
  could in principle echo path bytes if git ever returned a malformed line
  whose path field contained surprising characters. With `core.quotepath=false`
  paths come through raw rather than octal-escaped, so non-ASCII bytes would
  reach the log as-is. This is consistent with how the rest of the queue
  reports paths (and how `Fail` already includes captured stderr text on
  non-zero git exit), so I left it that way. Flagging in case the reviewer
  wants a sanitized-message variant.
- `T` is now in the alphabet, but the function does not treat `T` as
  rename/copy. That is correct — `T` is a same-path type change and the
  `hasRenameOrCopy` predicate (`statusCode.IndexOf('R') -ge 0 -or
  statusCode.IndexOf('C') -ge 0`) is unchanged, so a `T ` / ` T` / `MT` /
  `TM` record stays in the single-path branch. A combined `RT` or `CT`
  record would still enter the rename/copy branch via its `R`/`C`
  component, which is the desired behavior (both paths get allow-checked).
  Flagging only because it is an easy thing to second-guess in review.
