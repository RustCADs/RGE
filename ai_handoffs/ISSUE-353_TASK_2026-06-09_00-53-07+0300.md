## Deliverables

- A host-local persistence helper, either inside `menu.rs` or in optional `palette_recent.rs`, that stores only command diagnostic id strings and enforces the existing recent-list cap and most-recent-first deduplication.
- `EguiHost` construction loads persisted command-palette recents from a deterministic config path, with an injectable path or equivalent test seam for unit tests.
- Successful command-palette activations update the in-memory recent list and attempt to save it; persistence write failures are logged or ignored without blocking command dispatch.
- Main-menu activations continue to enqueue commands through the existing `MenuCommandHandoff` path and do not update or persist command-palette recents.
- Focused `rge-editor-egui-host` unit tests cover load/save, cap/dedup, stale persisted ids, disabled persisted ids, non-fatal missing/corrupt/unwritable persistence, main-menu non-recording, and unchanged non-blank fuzzy ordering.
- Planning/status docs are updated only if useful to record the dispatch outcome; no architecture, doctrine, Cargo, schema, workflow, runtime, kernel, plugin, or editor-shell/ui/actions/state documentation is required.

## Acceptance Criteria

- Persisted data is just the existing `Command::diagnostic_id()` values used by the command palette; no favorites feature, second command model, alternate action registry, or new command execution path is introduced.
- Blank command-palette filters promote only currently enabled persisted recent ids, ignore stale ids, and leave disabled matching ids in their normal original-order remainder position rather than promoting them.
- Non-blank command-palette filters preserve the existing fuzzy ranking order regardless of persisted recents.
- Missing, unreadable, corrupt, or unwritable recent-persistence files are non-fatal: the host still constructs or dispatches the selected command, and tests pin this behavior.
- Saving happens only after a successful command-palette activation that returns an enabled command; opening, closing, filtering, selection movement, disabled rows, and main-menu clicks do not record recents.
- The implementation stays within the allowed `editor-egui-host` source/test surface plus optional planning/status docs, and `Cargo.toml`, `Cargo.lock`, every `**/Cargo.toml`, schemas, workflows, dispatch automation, architecture-lint config, kernel, runtime, plugins, and `editor-shell`/`editor-ui`/`editor-actions`/`editor-state` files remain untouched.

- Do not add dependencies or touch Cargo metadata.

- Do not record recents from main-menu activations, accelerator handling, or any editor-shell route.

## Planner Notes

The key design constraint is to persist the already-existing recent activation ids, not to introduce a new command-palette substrate. Prefer a small helper with explicit load/save functions and tests that use a temporary path. If the deterministic runtime config path is platform-specific, keep it dependency-free and private to `editor-egui-host`; tests should avoid writing to the real user config path.
