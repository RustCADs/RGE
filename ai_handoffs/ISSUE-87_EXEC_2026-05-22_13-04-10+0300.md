## Central-Row Metric and Threshold

- **Metric:** central-row (`y = 128` on a 256×256 viewport)
  accumulated absolute per-channel RGB delta between the
  imported-NORMAL render and the flat-recompute render, summed over
  the subset of pixels that landed on the sphere in BOTH renders.
- **Measured value (local run):** `sum_abs_rgb_delta = 1062`,
  `common_on_mesh_pixels = 58`, `max_single_pixel_delta = 66`.
- **Committed threshold:** `sum_abs_rgb_delta > 300`. This is the
  Task Packet's named driver-noise floor (`sum_abs_rgb_delta > 300`
  rule of thumb); the local measurement clears it by ~3.5× margin.
  The companion `common_on_mesh_pixels > 8` gate fires before the
  delta gate so a render that ended up entirely off-sphere can't
  silently satisfy the delta with one well-lit pixel.

One implementation note worth surfacing for the Reviewer (not a
deviation): the sphere's triangle index buffer is sorted by
centroid `dot((1, 1, 1))` ascending so the visible hemisphere draws
LAST. This is necessary because the editor's lit-mesh pipeline
uses `depth_write_enabled: false`
(`editor-shell::render_path::lit_mesh_depth_state`); without the
sort, back-facing triangles overdraw front-facing ones at the
silhouette and the sphere renders as ambient-only (all
`(18, 18, 18)` pixels in early diagnostic runs). Fixing the
underlying depth state would require touching production
render-path code, which the Task Packet's `MUST NOT edit` envelope
forbids; sorting at fixture build time is the substrate-correct
workaround inside the allowed envelope.
