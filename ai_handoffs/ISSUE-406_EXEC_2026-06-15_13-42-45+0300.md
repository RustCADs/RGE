### Candidate 1: Minimal Render-Path Refresh After Phase-1-Only Init

Current render source supports the concern but makes it a broader render/GPU
feature than the smallest next task. `render_path.rs` documents Phase 1 as
unconditional window/GfxContext/surface/egui setup and Phase 2 as gated on
`has_cad_scene || has_prebuilt_mesh` (`render_path.rs:207-222`, `232-287`).
The Phase-2 helper sources CAD geometry via
`projection.render_mesh_for(entity, cad_world)` and uploads it with
`LitMesh::from_render_mesh` (`render_path.rs:975-1006`). The existing
`reload_render_assets` path is a render-asset/GPU swap path and rejects calls
before render state is initialized (`render_path.rs:1095-1127`, `1146`).

Conclusion: not selected. A post-Phase-1 CAD mesh refresh would require
render-path behavior, mesh-upload behavior, and GPU-state lifecycle decisions.
That is coherent future work, but it is not the smallest feature after the
first cuboid add and it would be easy to mix with product decisions about when
the add entry point becomes user-facing.

### Candidate 2: Bounded CAD Restore Semantics Cleanup

Current CAD-core source shows `CadGraph::restore_to` replaces the inner graph
from the checkpoint snapshot, sets the root only when `root_at_checkpoint` is
`Some`, and updates `history.head` (`crates/cad-core/src/checkpoints/mod.rs:347-360`).
ISSUE-405 tests avoid changing CAD-core semantics: they restore to
`pre_add_head`, assert zero nodes, then explicitly clean projection state with
`CadProjection::despawn_brep_entity` and a projection tick
(`lifecycle/tests.rs:2447-2480`; `crates/cad-projection/src/lib.rs:479`).

Conclusion: not selected. A true restore semantics cleanup either belongs in
`crates/cad-core/**` or needs a product/architecture decision about undo,
checkpoint authority, and projection cleanup ownership. Task 151 explicitly
forbids choosing that route unless it stays out of CAD core and preserves the
no-global-undo boundary.

### Candidate 3: Headless Editor-Shell Observability/Test Coverage

Current editor-shell state is observable only through private fields and test
helpers. The CAD fields are crate-private in `lifecycle/mod.rs:360-375`;
`current_scene_bounds` and `selected_cad_scene_bounds` are private at
`lifecycle/mod.rs:2069` and `lifecycle/mod.rs:2084`. Tests currently duplicate
private helper logic with `cad_brep_entities` and `cad_render_meshes`
(`lifecycle/tests.rs:2291-2311`). There is no existing
`cad_scene_inspection` / `CadSceneInspection` / `scene_inspection` API.

Command routing, shortcuts, save/load, and CommandBus surfaces are separate
existing paths, not needed for this candidate. Source references include
`route_menu_command` in `render_path.rs:415`, shortcut-to-menu routing in
`lifecycle/mod.rs:2936-2962`, save/open handlers in
`lifecycle/save_request.rs:205` and `lifecycle/open_request.rs:233`,
`EditorShell::submit_action` / `undo_command` / `redo_command` /
`mark_saved_command` in `lifecycle/commands.rs:407-436`, and
`CommandBus::submit` plus the `Action` trait in `crates/editor-actions/src`.

Conclusion: selected. A headless read-only CAD scene inspection snapshot can be
implemented entirely in `crates/editor-shell/src/lifecycle/{commands.rs,mod.rs,tests.rs}`.
It improves observability for the newly landed CAD add path, stays out of
render refresh / CAD core / cad-projection / CommandBus / save-load / UI
surfaces, and is independently shippable.
