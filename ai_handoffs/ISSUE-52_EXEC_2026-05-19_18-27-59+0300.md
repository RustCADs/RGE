## Per-File Summary

Root cause of the correction-round verification failure: the repository has
`core.autocrlf=true` and no `.gitattributes`. The HEAD blob and index blob of
`crates/material-graph/src/lib.rs` are already pure LF (verified: 0 CR bytes,
653 LF bytes in both `git show HEAD:` and `git cat-file -p :`). Because
autocrlf re-materializes files with CRLF on checkout under Windows, the
working-tree copy that `cargo +nightly fmt --all -- --check` reads was CRLF,
which conflicts with `rustfmt.toml`'s `newline_style = "Unix"` — producing
`Incorrect newline style in ...\crates\material-graph\src\lib.rs`.

The fix normalizes only the working-tree file to LF so the canonical fmt
gate reads Unix newlines. This is purely a line-ending change:
`git diff --ignore-cr-at-eol --stat crates/material-graph/src/lib.rs` reports
no content change, and `git diff --stat` (with autocrlf normalization)
likewise shows no semantic delta. No edits were made outside
`crates/material-graph/src/lib.rs`; git configuration was deliberately left
untouched as out of scope.

The `M crates/material-graph/src/lib.rs` working-tree status is benign: since
the tracked blob is already LF, an autocrlf-enabled `git add`/commit
normalizes the LF working copy back to the identical LF blob, so material-graph
contributes no committed delta — the ISSUE-52 commit remains the anim-graph
test-only change.
