## Typed scene-loader impact section

The recent typed `ComponentValue` and `rge-scene-loader` work **does not
adopt any reflection machinery**. It is an **explicit match-table /
typed-RON bridge** end-to-end:

1. **Wire shape stays reflection-neutral.** `ComponentValue { type_id:
   String, data: String }` (`crates/rge-data/src/scene.rs:84-95`) carries
   a canonical string identifier plus an opaque RON literal. Neither
   field is a `kernel/types::TypeId` (which would be the 128-bit FNV-1a
   hash from `kernel/types/src/type_id.rs:54-103`). The `type_id` field
   is just a `String` keying the loader's match arms.

2. **Loader bridges via a literal match table on the string, not a
   registry lookup.** `crates/rge-scene-loader/src/lib.rs:106-161`
   dispatches `component.type_id.as_str()` through four arms whose left
   sides are the literal strings `"rge::components::Transform"` /
   `"rge::components::Camera"` / `"rge::components::Light"` /
   `"rge::components::Visibility"`. Each arm calls `ron::from_str::<T>`
   with the concrete component type and `world.insert(ecs_id, value)` —
   no `dyn Reflect`, no `ReflectValue`, no `TypeId::of_name`, no
   trait-object handle, no `kernel/types::serde_bridge::from_ron`, no
   inventory/registry lookup. The crate manifest at
   `crates/rge-scene-loader/Cargo.toml:11-19` does not declare
   `rge-kernel-types` or `rge-macros-reflect`.

3. **Components are typed via plain `serde::{Serialize, Deserialize}`
   derives + `rge_kernel_ecs::Component`.** Each component's `Cargo.toml`
   pulls only `serde` (plus `rge-kernel-ecs` for the `Component` marker
   trait); each `*.rs` carries a normal `#[derive(... Serialize,
   Deserialize)]` and `impl rge_kernel_ecs::Component for T {}` (see
   `crates/components-spatial/src/transform.rs:22-31,59`,
   `crates/components-render/src/camera.rs:43-69`,
   `crates/components-render/src/light.rs:56-77`,
   `crates/components-visibility/src/visibility.rs:19-30`). Not a single
   component derives `Reflect` or imports `rge_kernel_types`.

4. **Production consumer is reflection-free.** `runtime-headless` runs
   `rge_scene_loader::load_scene_into_world` (`runtime/runtime-headless/src/main.rs:14,95`)
   and reaches no further into the reflection stack. Its manifest
   (`runtime/runtime-headless/Cargo.toml:16-19`) declares only
   `rge-data` + `rge-scene-loader` + `ron`.

5. **Compatible-but-not-adopted.** The `data` payload format would
   round-trip through `kernel/types::serde_bridge::from_ron::<T>` if a
   caller ever wanted to walk it through reflection, because both paths
   are RON over a `Serialize + Deserialize` type. The scene fixture's
   payload strings (`golden-projects/simple-scene/scenes/main.rge-scene:9-19,28-35`)
   are concrete RON-literal renderings of the typed components. The
   loader simply chose the explicit-match path over the reflection
   path — and crucially over a runtime-registry alternative that would
   either require `inventory!`-style global registration (forbidden by
   `kernel/types/src/lib.rs:40-43` and `kernel/types/BUDGET.md:50-52`)
   or a derive-emitted name→constructor map (not currently emitted by
   `crates/macros-reflect/src/codegen.rs`). The `crates/rge-data/src/lib.rs:36-40`
   and `crates/rge-data/src/scene.rs:14-25` doc-comments describe the
   two-phase "parse Scene then walk `data` through
   `kernel/types::serde_bridge::from_ron`" plan as **future** work —
   they predate the typed-RON loader and were not updated when the
   match-table choice landed.
