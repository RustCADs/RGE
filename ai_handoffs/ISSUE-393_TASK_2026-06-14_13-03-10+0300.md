## Goal

Implement GitHub issue #393 / task 139: on a qualifying viewport left double-click, frame the currently selected CAD projection entity bounds first, or the union of multiple selected CAD projection entity bounds when multiple selected CAD entities resolve. If no selected CAD bounds can be framed, preserve the existing scene-wide `reset_camera()` fallback. This is a narrow editor-shell camera/navigation polish after the viewport wheel zoom, orbit, pan, frame-all, drag-cancel, and cursor-leave work.

## Acceptance Criteria

- A qualifying viewport left double-click still uses the existing `ViewportLeftDoubleClick` timing/distance detector; do not edit or replace that detector.
- The first and second left presses still run the existing `should_fire_face_pick(egui_consumed, over_viewport_tab)` / `handle_left_click()` path exactly as before. The camera framing happens only after the existing face-pick handling on the qualifying second press.
- When the shell is on the CAD path and at least one selected entity resolves through `CadProjection::render_mesh_for(entity, cad_world)` to finite renderable bounds, the double-click frames the selected CAD mesh bounds with `isometric_camera_for_bounds`.
- When multiple selected CAD entities resolve to renderable bounds, the double-click frames the union of their selected bounds via `compute_aabb_union`; unselected CAD entities must not expand the selected-bounds camera target.
- If there is no selection, no selected entity resolves to renderable CAD bounds, selected bounds are non-finite/empty, required CAD projection state is absent, or the shell is in the prebuilt render-mesh path with no CAD entity mapping, the double-click falls back to the existing scene-wide `reset_camera()` behavior.
- `EditorShell::reset_camera()` itself remains scene-wide and unchanged for View menu / Home command routing. Do not make View camera commands frame selected entities.
- Wheel zoom, right-button orbit, middle-button pan, cursor-left/focus-loss drag cancellation, viewport hit testing, command routing, shortcut execution, command-palette behavior, save/load, undo/dirty state, clipboard state, and face selection semantics remain unchanged.
- No CAD/projection state is mutated by the selected-bounds framing path; it is read-only with respect to `cad_world`, `projection`, `cad_graph`, `CommandBus`, undo/dirty, and save/load authority.
- No new command, menu item, accelerator, route owner, public API, crate dependency, or Cargo metadata is added.

## Planner Notes

- `EditorCoord::selection` stores `rge_kernel_ecs::EntityId`, the same entity ID type consumed by `CadProjection::render_mesh_for`, so the selected-bounds path should not require a new ID mapping layer.
- Keep `reset_camera()` as the canonical scene-wide frame-all function. Prefer a small private helper used only by `handle_viewport_left_press` for "selected CAD bounds else reset_camera" so existing View command behavior stays load-bearing and testable.
- For fallback semantics, unresolved selected entities should never panic. If the selected renderable mesh collection produces no finite AABB, call `reset_camera()` exactly as the current double-click path does.
