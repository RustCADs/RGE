## Goal

Add automatic GLB hot-reload for GitHub issue #85. When `rge-editor` is launched
with `--glb PATH`, a notify-backed watcher should observe that active source
file's parent directory, coalesce write noise, and request a reload during the
redraw path. The watcher is only a producer of reload requests: actual reload
semantics, failure handling, warning logs, and render-asset mutation must remain
inside the existing `EditorShell::handle_asset_reload` path. This packet also
addresses the prior failed attempt's verification feedback by permitting only a
narrow newline-style normalization for
`kernel/graph-foundation/tests/diff_correctness.rs` if the canonical fmt gate
still reports that exact pre-existing failure.

## Deliverables

- A notify-backed GLB watcher owned by `rge-editor` and enabled only when
  `--glb PATH` is active.
- The watcher watches `glb_source_path.parent()` with
  `notify::RecursiveMode::NonRecursive` and keeps the underlying notify watcher
  object alive for the editor lifetime.
- The watcher filters to `notify::EventKind::Modify(_)` events that target the
  active `glb_source_path`. Sibling files, directories, and non-modify events
  must not request reload.
- The watcher debounces/coalesces file-write noise at roughly 200 ms before it
  exposes a reload request.
- Pending watcher requests are drained at `WindowEvent::RedrawRequested` time.
  The drain path MUST call `EditorShell::handle_asset_reload()` for the actual
  reload.
- The watcher and redraw-drain code MUST NOT call `load_all_glb_meshes`, call
  `reload_render_assets`, or mutate render asset fields directly. All reload
  semantics stay in `handle_asset_reload`.
- Existing manual R-key reload behavior continues to work.
- Focused coverage proves malformed-bytes writes retain the prior frame/state,
  the loader `Err` path warn-logs, and the same watcher remains live so a later
  valid write to the same path triggers a successful reload.
- If the prior newline-style fmt failure is still present, normalize only
  `kernel/graph-foundation/tests/diff_correctness.rs` line endings so the
  canonical fmt gate can pass.
- An `ISSUE-85_EXEC_*.md` execution report documenting changed files, test
  coverage, verification results, and any use of the newline-style exception.

## Acceptance Criteria

- `rge-editor --glb PATH` starts a GLB watcher for that exact source path;
  running without `--glb` starts no GLB watcher and preserves default cuboid
  behavior.
- The watch target is the source path's parent directory, watched
  non-recursively.
- Notify callback/drain logic treats the watcher as a producer of reload
  requests only. Reload work happens only because
  `EditorShell::handle_asset_reload()` is called from the redraw drain path.
- Multiple modify notifications from a single write are coalesced by an
  approximately 200 ms debounce so one save burst cannot produce unbounded
  reload calls.
- Modify events for unrelated sibling files do not call
  `handle_asset_reload()`.
- A malformed-bytes write to the watched GLB path causes a drained reload
  request, reaches the existing loader `Err` handling through
  `handle_asset_reload()`, emits a warning, and retains the previously
  rendered/loaded asset state.
- After that malformed write failure, the same watcher remains live; a
  subsequent valid GLB write to the same path triggers another drained request
  and successfully reloads through `handle_asset_reload()`.
- No implementation path in the watcher directly mutates `EditorShell` render
  assets, materials, mesh vectors, GPU state, or prebuilt render fields.
- No new crate/package is added, no dependency version is changed, and
  `Cargo.lock` is unchanged.
- If `kernel/graph-foundation/tests/diff_correctness.rs` is touched, its diff
  is newline-style normalization only and `cargo +nightly fmt --all -- --check`
  no longer reports the prior "Incorrect newline style" failure.
- The focused cargo tests, architecture lint gate, and canonical dispatch
  verification gate pass.

## Constraints / Non-Goals

- Do not redesign asset reload semantics. `handle_asset_reload` remains the
  single place that decides whether reload is allowed, invokes the loader hook,
  handles loader/render errors, emits warning logs, and retains the previous
  frame.
- Do not move GLB loading/import logic into `editor-shell`; `rge-editor`
  remains the crate that knows about `rge-io-gltf` and `load_all_glb_meshes`.
- Do not add a polling watcher, a timer thread that performs reload work, a
  recursive directory watch, or a dependency on existing RON/theme watcher
  modules.
- Do not broaden CLI behavior or add new CLI flags.
- Do not attempt cross-platform editor-save rename handling beyond the
  requested parent-directory notify watch, modify-event filtering, debounce,
  and redraw-time drain.
- Do not add tracked binary fixtures. Use existing GLB fixtures or temporary
  files created at test runtime.
- Do not fix unrelated rendering, importer, lifecycle, or test-flakiness
  issues.
- The newline-style exception is verification hygiene only; it does not
  authorize any kernel behavior change.

## Planner Notes

Existing R-key reload already has the needed semantics: loader errors warn and
retain prior state, while successful reloads flow through the atomic
`reload_render_assets` path. This dispatch should add only the missing
automatic trigger. A clean implementation can either wrap/delegate
`EditorShell`'s `ApplicationHandler` from `rge-editor` and intercept
`RedrawRequested`, or add a very small generic hook to `editor-shell` that is
invoked during `RedrawRequested`; prefer the approach with the smaller source
surface. In both cases, the notify callback should only set or send pending
reload intent and should keep running after both notify errors and loader
errors.

The prior ISSUE-85 attempt failed before control review because the canonical
verification gate stopped at `cargo +nightly fmt --all -- --check` with
`Incorrect newline style in kernel/graph-foundation/tests/diff_correctness.rs`.
That file is unrelated to GLB watching, so this packet authorizes only newline
normalization if the exact failure remains. Do not use that exception for any
other kernel change.
