## What I Changed

### Source

- `Invoke-AiDispatchLoop.ps1`:
  - Added `MutationRetryCount` parameter, `[ValidateRange(0, 5)]`, default
    `1`. Value `0` is the previous single-attempt path: no snapshot, no
    restore, no retry output.
  - Added `New-MutationSnapshot` / `Restore-MutationSnapshot` /
    `Remove-MutationSnapshot` helpers immediately after
    `Invoke-WithSamePhaseRetry`. Tracked changes (staged + unstaged) ride
    on a pinned `git stash create` commit under
    `refs/dispatch-snapshot/<safe-label>`. Untracked non-ignored files are
    captured into an in-memory `[ordered]` byte map via
    `git ls-files --others --exclude-standard`.
  - Added `Invoke-WithMutationRetry`: when `RetryCount > 0`, it captures
    one phase-entry snapshot, then for each attempt failure runs
    `git reset --hard HEAD && git clean -fd`, replays the stash with
    `git stash apply --index`, and rewrites the captured untracked
    manifest. Ignored paths (`.ai/dispatch-*/`, `*.log`, `/target/`) are
    deliberately left alone, so external caches and run-dir transcripts
    persist across retries. On exhaustion the wrapper clears
    `RetryableFailEnabled` and re-enters the canonical `Fail` path so the
    queue runner / scheduled task still sees the original terminal wording
    and exit behavior.
  - Wired the wrapper around three call sites:
    1. The `Invoke-ClaudeExecute` round-loop call — the `EXEC_STATUS`
       semantic check (`blocked` / `failed`) stays OUTSIDE the wrapper, so
       those verdicts are still terminal and never retried.
    2. The verification-failure `Invoke-CorrectionPacket` call — both the
       correction-packet write and the `Assert-PlannerScopeClean` guard
       run inside the retried action so an out-of-scope correction is
       reverted before the next attempt.
    3. The control-`needs_changes` `Invoke-CorrectionPacket` call —
       identical wrapping, identical scope-clean guarantee.
  - Side effect: normalized `git rev-parse --show-toplevel` output (which
    uses forward slashes on Windows) before composing untracked-manifest
    paths with `Join-Path`.

The existing same-phase retry around the two read-only model review phases
(`Invoke-ClaudePlanGate`, `Invoke-CodexControl`) is untouched. Plan-gate
and control `needs_changes` / `block` verdicts keep their current
behavior; only invocation / contract failures are retried. The new
parameter is range-validated identically to the existing retry knobs.

## Per-File Summary

Single-file change. The retry wrapper short-circuits on `RetryCount=0`,
returning the action's value with no snapshot or output, so callers in
the zero-retry path behave exactly as before. With `RetryCount > 0` the
phase-entry snapshot is captured once and reused across attempts.
Snapshot restoration is git-driven (`reset --hard`, `clean -fd`,
`stash apply --index`) plus an in-memory rewrite of untracked
non-ignored files; nothing outside the repository root is ever touched.

## Deviations from Task Packet

None — execution stayed strictly within the Task Packet scope.

Note on snapshot implementation choice: the TASK calls for tracked +
untracked-non-ignored preservation. A first cut used
`git stash create -u` and `git stash apply --index` for both. A
focused-harness probe revealed that Git for Windows' `git stash apply`
does NOT extract the untracked parent tree (exit 0, no error, untracked
files silently missing). The fix is split capture: tracked changes ride
on `git stash create` (which `git stash apply --index` does restore
correctly), and untracked non-ignored files are spooled to an in-memory
byte map. This still preserves "the exact phase-entry tracked state plus
the exact phase-entry set and contents of untracked non-ignored files"
required by the TASK, but does it via a path that actually works on this
platform. No semantic dispatch outcome was reclassified as retryable.
