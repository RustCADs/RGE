## Goal

Implement GitHub issue #198 by adding snapshot-backed same-phase retry support for the two dispatch-loop mutation phases that can edit the repository: Claude execution and Codex correction-packet creation. The retry must protect pre-existing and in-progress work by restoring the exact phase-entry tracked state plus untracked non-ignored files before each retry attempt, while leaving all semantic dispatch outcomes and queue/publish behavior unchanged.

## Deliverables

- Add a mutation retry control path in `Invoke-AiDispatchLoop.ps1` that applies only to the phases invoking `Invoke-ClaudeExecute` and `Invoke-CorrectionPacket`.
- Add a user-visible zero-retry path, such as a bounded `MutationRetryCount` parameter with `0` supported, where mutation behavior remains the previous single-attempt behavior: no retry, no restore attempt after failure, and no extra retry success or exhaustion output.
- Capture a phase-entry snapshot before each retry-enabled mutation phase. The snapshot must preserve tracked file state, including staged and unstaged tracked changes if present, plus untracked non-ignored files and their contents.
- Restore the captured phase-entry state before every mutation retry attempt. Restoration must remove files created by the failed attempt, undo tracked edits/deletions from the failed attempt, and recreate any phase-entry untracked non-ignored files.
- Wire snapshot-backed retry around Claude execution rounds only at the `Invoke-ClaudeExecute` call site. Do not include verification, EXEC_STATUS semantic handling, or Codex control in that retry scope.
- Wire snapshot-backed retry around correction-packet creation at both existing correction call sites. Include the existing planner scope-clean check in the retried mutation phase so a failed or out-of-scope correction attempt is restored before retrying.
- Emit human-readable loop output for mutation retry failure, worktree restore before retry, retry success, and retry exhaustion. Use ordinary loop text output only; do not add or change schemas, JSONL trace formats, or structured log contracts.
- Preserve the existing read-only same-phase retry behavior for Claude plan gate and Codex control. Do not convert semantic verdicts into retryable errors.
- Produce the ISSUE-198 EXEC packet with a concise implementation summary, restore-harness result, verification results, and final git status.

## Acceptance Criteria

- `Invoke-ClaudeExecute` receives snapshot-backed retry for retryable invocation/contract failures only. A returned `EXEC_STATUS: blocked` or `EXEC_STATUS: failed` remains terminal for that round and is not retried.
- `Invoke-CorrectionPacket` receives snapshot-backed retry for retryable mutation failures, including failures caught by the planner scope-clean guard after the correction packet writer returns.
- Verification failure remains a correction-round input or terminal failure according to existing `MaxCorrectionRounds` behavior; it is not retried as a same-phase mutation failure.
- Codex control `needs_changes` and `block` verdicts keep their current behavior and are not retried as failures.
- Claude plan-gate `needs_changes` and `block` verdicts keep their current behavior and are not retried as failures.
- `MutationRetryCount` or the chosen equivalent is range-limited, defaults to enabling one mutation retry, and supports `0` as a previous-behavior single-attempt path.
- With zero mutation retries configured, a failed Claude execution invocation or failed correction-packet mutation takes the same single-attempt failure path as before this dispatch, aside from unavoidable code organization changes.
- Before each mutation retry attempt, the repository worktree is restored to the exact phase-entry tracked state plus the exact phase-entry set and contents of untracked non-ignored files.
- Ignored files and external caches are not treated as part of the required snapshot unless the implementation already manages a dispatch-owned output path explicitly.
- Retry cleanup is path-safe: use literal paths, repo-relative manifests, and git commands scoped to the repository root; do not delete outside the repository.
- Existing queue publishing, GitHub issue comments, label behavior, failure taxonomy, JSONL schemas, verification gates, Rust/Cargo behavior, docs, CI, scheduler behavior, and task-selection brief content are unchanged.
- Loop output contains distinguishable text for retryable mutation attempt failure, successful restore before retry, successful retry completion, and retry exhaustion.
- The canonical verification gate passes after the implementation.

## Planner Notes

The existing loop already has same-phase retry around read-only model review phases. This dispatch is narrower: add snapshot-backed retry only where a retry could otherwise leave repository mutations behind. The safest shape is a separate mutation retry wrapper that captures a worktree snapshot at phase entry, restores it before each retry, and delegates final failure back to the existing `Fail` path so terminal wording and exit behavior remain familiar.

The focused restore harness should avoid permanent test files. A practical approach is to create a temporary git repository outside the project tree, run the new helper or a minimal fake-command dispatch path against that temporary workspace, assert status/content before the retry attempt, and clean up after itself. If the harness uses fake `codex` or `claude` commands, keep them in the temporary workspace or system temp directory and do not leave them in this repository.
