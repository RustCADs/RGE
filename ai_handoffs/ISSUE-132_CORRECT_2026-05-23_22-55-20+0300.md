   production caller can make `TimeScale` non-default today. From a normal

   a genuinely user-visible World-only `CommandBus::submit` action that

   the corrected EXEC report must follow the TASK packet's `NEEDS_HUMAN`
   route. Acceptance: Q2 no longer claims a fresh-editor `Ctrl+0` reset is a
   valid submitting action unless the report cites a production non-defaulting
   path; Q3 describes only the corrected Approach A implementation shape; Q5
   names one bounded follow-up task only if the non-noop, user-visible,
   World-only submit path is evidenced, otherwise Q5 says `NEEDS_HUMAN` and
   names the smallest human decision needed.
2. **Correct the production `Action` inventory** - Codex control finding: the
   EXEC report states `SetTimeScale` is the only production `Action` impl, but
   `crates/editor-actions/src/compound.rs` contains production
   `CompoundAction` and `impl Action for CompoundAction`. Required change:
   update the Q1/source inventory and any command-log conclusions so they do
   not claim `SetTimeScale` is the only production `Action` implementation.
   Classify `CompoundAction` accurately as infrastructure/non-candidate if it
   has no direct user-visible production trigger in the audited roots.
   Acceptance: the corrected EXEC report explicitly names `CompoundAction`,
   cites `crates/editor-actions/src/compound.rs`, and distinguishes production
   infrastructure actions from eligible user-visible candidates.

   report claims shell `rg` availability and `rg --files` usage, while the
   review environment reports `rg` unavailable and the report appears to have
   used Claude Grep/Glob tooling for several searches. Required change: make
   the command log/tooling section accurate. Do not claim `command -v rg`,
   shell `rg`, or `rg --files` ran unless they actually ran in the Executor's
   shell. If Claude Grep/Glob tooling was used, label it as tool-level search,
   not shell `rg`. If additional read-only searches are needed for the
   corrected report, use PowerShell fallbacks such as `Get-ChildItem` and
