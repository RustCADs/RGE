## Goal

Add an opt-in Codex pre-flight pitfall audit to `Invoke-AiDispatchLoop.ps1` for
GitHub issue #184. When explicitly enabled, the loop should ask Codex to review
the finalized TASK packet in a read-only sandbox before Claude execution, save a
validated checklist as `codex.preflight.md`, and surface that checklist to
Claude and Codex control as a non-authoritative aid. Existing dispatch behavior
must be unchanged unless the new switch is supplied.

## Deliverables

- Add a top-level switch parameter named `EnablePreflightAudit` to
  `Invoke-AiDispatchLoop.ps1`. Its default must be off, and existing
  invocations that do not pass the switch must not run the audit, write
  `codex.preflight.md`, or add any pre-flight audit text to Claude or Codex
  prompts.
- When `EnablePreflightAudit` is supplied for an execution-capable run, run one
  Codex pre-flight audit after the approved TASK packet is available and before
  Claude execute round 0. This applies to both normal plan-then-execute runs
  and `-ResumeApprovedTask` runs. `-PlanOnly` must retain its current
  stop-after-finalized-TASK behavior and must not run the audit.
- Run the pre-flight Codex audit with a read-only Codex sandbox. The audit may
  inspect the finalized TASK packet and repository context needed to identify
  scope-preserving pitfalls, but it must not edit files, stage, commit, push, or
  require any file outside the allowed implementation surface.
- Write the successful extracted audit artifact to exactly
  `.ai/dispatch-$DispatchId/codex.preflight.md`. Do not create or overwrite
  that file until extraction and validation of the current audit output
  succeeds. Do not use a stale pre-existing `codex.preflight.md` as the
  checklist for a new run.
- Use this required Pre-flight Audit Markdown shape in `codex.preflight.md`:

```markdown
# Pre-flight Audit

TASK_PACKET: ai_handoffs/ISSUE-184_TASK_2026-05-25_21-31-00+0300.md

## Boundary

The TASK packet remains authoritative. This checklist is advisory and must not
expand scope.

## Pitfall Checklist

- [ ] P1: Scope-preserving pitfall text.
- [ ] P2: Scope-preserving pitfall text.

## Verification Checklist

- [ ] V1: Verification focus text.
- [ ] V2: Verification focus text.
```

- The extracted checklist IDs must be stable P-number and V-number IDs:
  `P1`, `P2`, and so on for pitfalls, and `V1`, `V2`, and so on for
  verification focus items. IDs must be unique within the audit, must not be
  regenerated during prompt injection, and must remain visible wherever the
  checklist is injected.
- Validate extraction before writing `codex.preflight.md`. At minimum,
  extraction succeeds only when the current Codex output contains the required
  Markdown shape, has a `## Pitfall Checklist` section with one or more
  `- [ ] Pn:` items, has a `## Verification Checklist` section with one or
  more `- [ ] Vn:` items, and has no duplicate P-number or V-number IDs.
- If the pre-flight Codex command fails or extraction/validation fails, fail the
  dispatch before Claude execution with a clear message naming the pre-flight
  audit log or extraction failure. Do not write a partial
  `codex.preflight.md`, do not inject a stale checklist, and do not continue as
  though the requested opt-in audit succeeded.
- Inject the extracted checklist into the Claude execute prompt only for round
  0, only when a current checklist exists, and without changing the
  `EXEC_STATUS` / `EXEC_PACKET` marker contract. Do not inject it into Claude
  correction-round execute prompts.
- Inject the extracted checklist into the Codex control prompt on every control
  round when a current checklist exists, without changing
  `.ai/codex_control.schema.json` or the required JSON response shape.
- In both the Claude execute prompt and the Codex control prompt, state
  explicitly that the TASK packet remains authoritative and that the pre-flight
  checklist is advisory, must not expand scope, and must be ignored if it
  conflicts with the TASK packet.
- Do not modify `Invoke-CodexPrompt`; reuse it as-is or add separate helper
  logic around it. Do not modify the stall watchdog, timeout behavior, or any
  schema.

## Acceptance Criteria

- The only source file changed is `Invoke-AiDispatchLoop.ps1`.
- With `EnablePreflightAudit` omitted, behavior is unchanged: no pre-flight
  Codex audit command runs, no `codex.preflight.md` is written, no checklist is
  injected into Claude execute prompts, and no checklist is injected into Codex
  control prompts.
- With `EnablePreflightAudit` supplied for a normal execution run, the audit
  runs exactly once after TASK approval/finalization and before Claude execute
  round 0.
- With `EnablePreflightAudit` supplied for `-ResumeApprovedTask`, the audit
  runs exactly once after the approved TASK is located and before Claude execute
  round 0.
- With `-PlanOnly`, the run still stops after the approved TASK and does not
  run the pre-flight audit or write `codex.preflight.md`.
- The pre-flight Codex audit is invoked with `--sandbox read-only`.
- `codex.preflight.md` is written only after current-output extraction and
  validation succeed. Failed extraction does not leave a partial checklist file
  and cannot cause stale checklist injection.
- The audit Markdown uses the required `# Pre-flight Audit`,
  `## Boundary`, `## Pitfall Checklist`, and `## Verification Checklist`
  headings, and all checklist items keep unique stable `Pn` or `Vn` IDs.
- Claude execute round 0 receives the checklist only when it exists and sees a
  prompt statement that the TASK packet remains authoritative and the checklist
  must not expand scope.
- Claude execute rounds greater than 0 do not receive the pre-flight checklist.
- Codex control receives the checklist on every round when it exists and sees a
  prompt statement that the TASK packet remains authoritative and the checklist
  must not expand scope.
- The Codex control response schema and `.ai/codex_control.schema.json` remain
  unchanged.
- `Invoke-CodexPrompt`, `Invoke-WithTimeout`, the stall watchdog,
  `CodexStallThresholdSec`, timeout exit codes, Claude marker parsing,
  verification behavior, queue behavior, auto-driver behavior, scheduler
  behavior, Rust/Cargo files, docs, and schemas remain unchanged.
- The implementation remains compatible with Windows PowerShell 5.1 and does
  not use PowerShell 7-only syntax or APIs.

## Planner Notes

Prefer a small helper in `Invoke-AiDispatchLoop.ps1` that runs the read-only
Codex audit, extracts a marker-delimited Markdown body from the current audit
output, validates the required headings and IDs, writes
`codex.preflight.md` only after validation, and returns the checklist text for
conditional prompt injection.

The prompt text injected into Claude and Codex should be concise. Preserve the
P-number and V-number IDs so control can refer to them clearly, but make the
boundary statement hard to miss: the TASK packet remains authoritative, and the
checklist cannot authorize edits or tests outside the TASK packet.

Do not recursively test this by launching another full dispatch unless a human
explicitly asks for that. The required verification for this dispatch is the
PowerShell parser check, `git diff --check`, and the canonical
`.ai/dispatch.verify.ps1` gate.
