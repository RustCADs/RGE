### Q2. What is the current rge-data project and scene schema surface?

The `rge-data` crate is the RON serde+migration layer (see
`crates/rge-data/src/lib.rs` rows 1-50 module docstring; "stateless serde +
migration layer", no PIE state). The container shapes a scaffold must obey:

- **`Project`** (`crates/rge-data/src/project.rs` rows 27-44): fields
  `version: SchemaVersion`, `name: String`, `description: String`,
  `target_tiers: Vec<TargetTier>`, `plugins: Vec<PluginRef>`,
  `scenes: Vec<ScenePath>`. No additional `schema_version` field.
- **`ScenePath`** (`crates/rge-data/src/project.rs` rows 89-103):
  `#[serde(transparent)] pub struct ScenePath(pub String);` — on-disk RON is
  the bare string `"scenes/foo.rge-scene"`. Convention: project-root-relative
  path. Doc comment (rows 42-43): *"The editor loads the first entry as the
  start scene unless overridden by user prefs."*
- **`TargetTier`** (`crates/rge-data/src/project.rs` rows 63-74):
  `#[serde(rename_all = "lowercase")]` enum `Desktop | Mobile | Web |
  Headless` — RON wire form `desktop`/`headless`/etc.
- **`PluginRef`** (`crates/rge-data/src/project.rs` rows 79-87): `id:
  String`, `version_req: String`.
- **`Scene`** (`crates/rge-data/src/scene.rs` rows 33-45): `version:
  SchemaVersion`, `name: String`, `entities: Vec<Entity>`, `root_entities:
  Vec<EntityId>` — IDs of entities with no `ChildOf` parent.
- **`Entity`** (`crates/rge-data/src/scene.rs` rows 68-81): `id: EntityId`,
  `name: String`, `components: Vec<ComponentValue>`, `relations:
  Vec<Relation>`.
- **`ComponentValue`** (`crates/rge-data/src/scene.rs` rows 89-95):
  reflection-neutral envelope of `type_id: String` (e.g.
  `"rge::components::Transform"`) and `data: String` (a RON literal). Two-
  phase deserialize: the scene parses first, then a downstream caller would
  walk `data` through reflection-aware bridging — see lib.rs rows 36-39 and
  the **`Reflect` stub** note (lib.rs rows 70-80): *"This crate never
  invokes any methods on a `Reflect` impl — the trait only exists so
  downstream calling code can refer to a stable name when staging
  integration. Once W02 lands, replace this module with `pub use
  rge_kernel_types::Reflect;`."* This is load-bearing for Q3/Q4 — the
  `type_id` strings in the on-disk fixtures are placeholder paths, not
  references to registered runtime components.
- **`Relation`** (`crates/rge-data/src/scene.rs` rows 99-115): enum
  `ChildOf { parent: EntityId }` | `LinkedTo { kind: String, target:
  EntityId }`.
- **`EntityId`** (`crates/rge-data/src/entity_ref.rs` rows 36-57):
  `#[serde(transparent)]` ULID; serializes as the 26-char Crockford base32
  string (see `tests/fixtures/sample_scene.rge-scene` rows 6, 17, 27 for the
  on-disk form `"0000000000000G000000000000"`). `Display` is the lossy
  `e_<8 hex>` form for diagnostics only.
- **`SchemaVersion`** (`crates/rge-data/src/schema_version.rs` rows 60-71):
  `(major: u8, minor: u8, patch: u8)`; wire form is a quoted dotted-triple
  string `"0.1.0"` per the visitor in this module.
- **Migration registry**
  (`crates/rge-data/src/migration.rs` rows 53-100,
  `crates/rge-data/tests/schema_validation.rs` rows 153-177): each
  `FileKind` (`Project` / `Scene` / `Prefab`) carries a v0.0 → v0.1 chain;
  current schema target is `SchemaVersion::V0_1_0`.

The **canonical pretty-print configuration** that defines byte-identical
RON output is duplicated in two places that the W14 wave deliberately keeps
in lockstep: `crates/rge-data/examples/bake_fixtures.rs` rows 19-27 and
`crates/rge-data/tests/round_trip.rs` rows 27-35 — `depth_limit(64)`,
`new_line("\n")`, `indentor("    ")`, `struct_names(true)`,
`separate_tuple_members(false)`, `enumerate_arrays(false)`, plus a trailing
newline appended in `ser` (rows 31-34 / rows 37-43). Any future scaffold
that needs byte-identical round-trip MUST emit through this same config.

The **canonical fixture-driven contract** for a scene is encoded as: two
entities (one `Camera` with a `Transform` `ComponentValue`, one
`ChildLight` with `ChildOf` relation to the camera and its own `Transform`),
`root_entities: [parent]` — see `crates/rge-data/examples/bake_fixtures.rs`
rows 62-95 and the on-disk `crates/rge-data/tests/fixtures/sample_scene
.rge-scene` rows 1-35. The fixture's `Transform` data is a string-encoded
RON literal `"(translation:(...),rotation:(...),scale:(...))"` but is **never
parsed into a typed Transform inside this crate**; nothing downstream of
`rge-data` consumes the `data:` payload today (see Q3).

   camera, and light render"* to *"basic schema load only"* — and that
   downscope is a **golden-project policy decision**, not a mechanical
   scaffold. It also requires choosing where the smoke test lives (under
   `crates/rge-data/tests/`? under a new `tests/golden-projects/`
   harness? under `tools/ci/`?), which is itself a layout decision the
   audit has no authority to make.

Per the TASK packet's instruction (*"If the next step is a design or
golden-project policy choice, Q4 states the decision required instead of

  live — a new `tests/golden_projects.rs` integration test under
  `crates/rge-data/`, a dedicated `tools/ci/` script (today

  line 16 references *"dedicated CI workflows in `tools/ci/`"* that do
  not yet exist on disk.

already-implied bounded scaffold), Q4 does **not** name allowed files,
