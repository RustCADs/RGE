## Goal

Implement GitHub issue #345 by adding host-local, in-memory recent-command
ordering to the existing `editor-egui-host` command palette. Palette
activations should record the activated `Command::diagnostic_id()` in a
bounded most-recent-first list, and blank command-palette filters should
promote currently available recent enabled commands ahead of the remaining
projected entries while preserving the fuzzy non-blank filtering and scoring
shipped by task 98 / ISSUE-342.

## Deliverables

- Add a host-local recent-command id list to `EguiHost`, stored only in memory and never persisted. Use a named constant cap of 16 entries unless a more specific existing local palette cap is found.
- Record only successful command-palette activations: when `command_palette_window` returns an enabled row's `Command` and `lib.rs` enqueues it through the existing `MenuCommandHandoff`, record that command's `diagnostic_id()` string. Do not record palette opens, closes, filter edits, disabled rows, or main-menu activations.
- Maintain the recent list as most-recent-first. Recording an id already in the list must move that id to the front instead of creating a duplicate, then truncate to the cap.
- Extend the existing palette projection/filter helper path so a blank or whitespace-only filter returns available recent enabled commands first, in recent-list order, followed by every remaining projected entry in its existing projection order. Recent entries must not appear twice.
- Ignore stale recent ids that do not match any current projected entry. Disabled recent entries must not be promoted, but they must remain visible in the original-order remainder if they are still projected.
- Preserve non-blank command-palette behavior from task 98 / ISSUE-342: exact word/field, prefix, substring, fuzzy ordered-subsequence matching, score ordering, multi-term AND semantics, and original-order tie-breaks must remain unchanged by recent history.
- Keep activation on the existing menu projection and `MenuCommandHandoff` path. Do not add a second command model, second command registry, or new command dispatch path.
- Add focused tests in `crates/editor-egui-host/src/menu_tests.rs` for bounded recent-list de-duplication, stale-id ignoring, blank-filter recent ordering, and unchanged non-blank fuzzy ordering when recent history is present.
- Update only the forward status/planning records needed to say ISSUE-345 added host-local in-memory recent-command ordering, keeping prior fuzzy-search and remaining non-goal wording intact.
- Produce this dispatch's EXEC packet with changed files, implementation summary, test/gate results, deviations, and final git status.

## Acceptance Criteria

- The recent-command store contains diagnostic id strings from `Command::diagnostic_id()`, not `Command` values, labels, shortcuts, or persisted records.
- The recent-command store is host-local memory owned by `editor-egui-host`; it is not serialized, written to disk, moved to `editor-shell`, or exposed as plugin runtime/discovery state.
- The cap is deterministic and tested. Recording more than the cap leaves exactly the most recent capped set in most-recent-first order.
- De-duplication is deterministic and tested. Re-activating an existing id moves it to the front without increasing list length.
- Blank and whitespace-only filters use recent ordering. Current enabled entries whose diagnostic ids appear in the recent list are promoted first, ordered by recent-list position.
- Stale recent ids are ignored for ordering and never produce missing rows, duplicate rows, panics, or placeholder entries.
- Disabled entries whose ids are recent are not promoted ahead of enabled rows; they remain present in the unpromoted original-order remainder.
- All entries not promoted by recency remain in the exact existing projected order produced by `command_palette_entries()`.
- Non-blank filters ignore recent ordering entirely and keep the task 98 / ISSUE-342 fuzzy score ordering. A recent entry must not jump ahead of a better non-blank fuzzy/exact/prefix/substring match because of history.
- Existing tests for label/shortcut/diagnostic-id matching, multi-term matching, fuzzy-only matches, exact/prefix/substring outranking fuzzy, fuzzy compactness ordering, keyboard selection, selected-row affordance, disabled-row visibility, and selected command dispatch remain passing or are changed only to accommodate the explicit blank-filter recent-order behavior.
- `lib.rs` records recents at the palette activation enqueue point only; main-menu clicks continue to enqueue commands without changing recent palette history.
- `menu.rs` remains the host projection/filtering surface. Do not create new modules or new files for this work.
- Documentation/task-record updates, if made, are forward-only and limited to ISSUE-345 behavior and verification results.

## Planner Notes

Keep the implementation local and boring. The expected shape is a small
`Vec<String>` or `VecDeque<String>` field on `EguiHost`, a testable helper in
`menu.rs` for recording/reordering by diagnostic id, and a `lib.rs` call that
records the returned palette command immediately before or after pushing it
through `MenuCommandHandoff`. If helper signatures change, keep the old
two-argument non-blank filtering behavior available through tests or adapt the
callers so existing fuzzy tests still prove the same ordering.

For blank filters, think in two passes: first resolve recent ids to current
enabled projected rows without duplicates, then append every projected row not
already emitted in original order. This naturally ignores stale ids and keeps
disabled or non-recent entries visible in the remainder.
