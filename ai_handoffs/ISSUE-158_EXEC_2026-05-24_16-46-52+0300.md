### Q2. Exact RON payload strings

Current code justifies these candidate raw `ComponentValue.data` RON
strings (no struct-name prefix, matching the existing rge-data scene
fixture convention):

  `(translation:(0.0,0.0,5.0),rotation:(0.0,0.0,0.0,1.0),scale:(1.0,1.0,1.0))`

  rge-data scene fixture already uses the exact no-struct-name `Transform`

  position; this stays inside what current code already round-trips.

- `Camera` derives `Serialize` + `Deserialize` with fields
  `projection: Projection`, `viewport: [f32; 4]`, `priority: i32`,
  `is_active: bool` (`crates/components-render/src/camera.rs:43-66`). The

### Q3. ComponentValue.type_id strings

Current code proves the canonical `Transform` string
`rge::components::Transform`, but it does **not** directly prove canonical
strings for `Camera`, `Light`, or `Visibility`.

Proven canonical string:

- `Transform`: `"rge::components::Transform"`

Candidate strings by the existing `"rge::components::TypeName"`
convention:

- `Camera`: `"rge::components::Camera"`
- `Light`: `"rge::components::Light"`
- `Visibility`: `"rge::components::Visibility"`

Evidence and risk: `ComponentValue` is a reflection-neutral envelope with
fields `pub type_id: String` and `pub data: String`; the docblock pins
the naming convention as `"rge::components::Transform"`,
`"rge::components::MeshRenderer"`, … and explicitly comments
`// Canonical type path (matches kernel/types::TypeId::path)`
(`crates/rge-data/src/scene.rs:83-95`). The internal scene fixture and
prefab/exposed-override fixtures all use `"rge::components::Transform"`
verbatim (`crates/rge-data/src/scene.rs:132-135`,
`crates/rge-data/src/prefab.rs:76-80`,
`crates/rge-data/src/prefab.rs:99-111`,
`crates/rge-data/tests/round_trip.rs:75-90`,
`crates/rge-data/tests/round_trip.rs:111-123`,
`crates/rge-data/tests/fixtures/sample_scene.rge-scene:9-22`). The
component crates export the Rust types as `Transform`, `Camera`, `Light`,
and `Visibility`
(`crates/components-spatial/src/lib.rs:32-36`,
`crates/components-render/src/lib.rs:31-40`,
`crates/components-visibility/src/lib.rs:25-28`), so the
convention-based candidate strings are well-formed Rust paths, but the
current crate names are `rge-components-spatial`,
`rge-components-render`, and `rge-components-visibility` — not
`rge::components`. The `rge::components` namespace is therefore an
rge-data envelope convention rather than a Rust module path.

The blocking ambiguity is that `kernel/types::TypeId` is a 128-bit
FNV-1a-128-style hash of a fully-qualified type-name string, constructed
via `TypeId::of_name(name)` over
`module_path!() ++ "::" ++ stringify!(Ident)`
(`kernel/types/src/type_id.rs:48-73`,
`kernel/types/src/type_id.rs:114-131`). `Reflect` separately exposes both
a `const TYPE_ID: TypeId` and a `const FQ_TYPE_NAME: &'static str`
(`kernel/types/src/reflect.rs:68-84`). The derive macro currently fills
`FQ_TYPE_NAME` from
`::core::concat!(::core::module_path!(), "::", ty_name_lit)` while
emitting `TYPE_ID` as `TypeId::from_bytes([0u8; 16])` for Phase 1.1,
citing that `blake3` is not yet `const fn`
(`crates/macros-reflect/src/codegen.rs:96-117`). The same macro rejects
enums (`crates/macros-reflect/src/derive.rs:35-43`). None of `Transform`,
`Camera`, `Light`, or `Visibility` derives `Reflect` today — they derive
only `Serialize`/`Deserialize`
(`crates/components-spatial/src/transform.rs:22`,
`crates/components-render/src/camera.rs:43`,
`crates/components-render/src/light.rs:56`,
`crates/components-visibility/src/visibility.rs:19-20`); and `Visibility`
is an enum, which the current `Reflect` derive would reject regardless.

Because `ComponentValue.type_id`'s canonical-string convention is
anchored in rge-data fixture text rather than in a derive-emitted
constant, and because the `kernel/types::TypeId` hash surface is the only
place that formalizes a fully-qualified type-name string (and that path
would yield `rge_components_spatial::transform::Transform` rather than
`rge::components::Transform`), the `Camera`, `Light`, and `Visibility`
strings remain plausible candidates rather than proven canonical strings.

### Q4. Raw payload fixture safety and schema-only tests

Raw `ComponentValue` payloads are schema-only safe in the narrow sense:
`rge-data` parses `ComponentValue.data` as an inert `String`, and the
`scene` module documents a two-phase model where callers parse the scene
first and only later walk `ComponentValue::data` through
`kernel/types::serde_bridge::from_ron` keyed by `ComponentValue::type_id`
(`crates/rge-data/src/scene.rs:17-24`,
`crates/rge-data/src/scene.rs:83-95`,
`crates/rge-data/src/lib.rs:34-49`,
`crates/rge-data/src/lib.rs:70-80`). The current load+tick harness
intentionally ignores component payloads
(`crates/rge-data/tests/golden_simple_scene_load_tick.rs:11-14`,
`crates/rge-data/tests/golden_simple_scene_load_tick.rs:50-57`), and the
existing fixture round-trip test already proves byte-identical RON →
struct → RON for `ComponentValue { type_id, data }` envelopes
(`crates/rge-data/tests/round_trip.rs:139-147`).

Schema-only tests that would be safe — after a human confirms the
type-id strings — are:

- Extend `crates/rge-data/tests/golden_simple_scene_schema.rs` to assert
  the expected `ComponentValue` count per entity and, for each, the
  expected `type_id` string and the verbatim raw `data` string.
- In the same test file, parse each raw `data` string with `ron::Value`
  (already a workspace dependency through `rge-data`) to catch malformed
  RON without pulling any component-crate dependency into `rge-data`'s
  dev-deps.
- Keep `crates/rge-data/tests/golden_simple_scene_load_tick.rs` strictly
  identity-only; it asserts entity-count parity and `advance_tick`
  behavior and does not translate component data
  (`crates/rge-data/tests/golden_simple_scene_load_tick.rs:60-105`).

Those tests would not require typed component bridging, the eventual
`kernel/types::serde_bridge` wiring, asset loading, cook output,
screenshot baselines, renderer/GPU behavior, editor UI, a new workspace
dependency, or any source-loader changes. They would still require edits
to test files and the golden-project scene fixture, which are out of
scope for this read-only dispatch.
