### Q1. Current Contract

**CommandBus (`crates/editor-actions/src/bus.rs`).** `pub struct CommandBus { stack: UndoStack, coalesce: CoalesceWindow, ledger: AuditLedger, diagnostics: DiagnosticAggregator }`. The submit signature is `pub fn submit(&mut self, action: Box<dyn Action>, world: &mut World) -> Result<(), BusError>` with `use rge_kernel_ecs::World;` — it still takes a mutable kernel ECS `World` as the sole mutation target. Undo / redo also take `&mut World`. The bus owns its own audit ledger (`rge_kernel_audit_ledger::AuditLedger`) and applies, projects, and pushes in lockstep on each submit. Coalescing within a 500 ms window collapses same-`ActionId` submits without growing the stack.

**Action trait (`crates/editor-actions/src/action.rs`).** `pub trait Action: Send + Sync + 'static` with `fn apply(&self, world: &mut World) -> Result<(), ActionResult>` and `fn revert(&self, world: &mut World) -> Result<(), ActionResult>`, plus optional `merge(&mut self, next: &dyn Action) -> MergeOutcome` and `payload(&self) -> Vec<u8>`. ### Q3. Command-Context Shape

Three approaches considered, each evaluated against the five axes named in the TASK packet.

**A. Keep the existing World-only `Action` surface unchanged.**

- *Dependency boundaries.* `editor-actions` keeps its current `Cargo.toml` (deps: `rge-kernel-ecs`, `rge-kernel-audit-ledger`, `rge-kernel-diagnostics`, plus `serde`, `thiserror`, `tracing`). No cross-tier dep changes. Editor-actions stays kernel-adjacent. Cleanest from the dependency-graph perspective; no `forbidden-dep` lint surface to renegotiate.
- *Ownership authority.* `World` is the sole mutation surface the bus controls. `CadGraph` (in `cad-core`) and `editor-state` coordination structs remain outside the bus's authority. Mutations to them happen out-of-band (e.g. `handle_left_click` writes `self.coord.face_selection` directly; `CadGraph` mutations require `begin_operation`/`graph_mut`/`commit` against `EditorShell.cad_graph`).
- *Audit / undo impact.* Bus audit-ledger (`rge_kernel_audit_ledger::AuditLedger` inside `CommandBus`) records one `EventKind::Action` per submit + truncation on redo-tail trim. `UndoStack` cursor moves in lockstep. For any user action whose effect is **not** representable as a `World`-component / `World`-resource mutation, the action either cannot be undone via the bus at all, or it must encode the side-effect as a `World` mutation (e.g. mirror the editor-state into typed components). The CAD checkpoint history (`CadGraph::CheckpointHistory`) remains a *separate* undo timeline; the bus cannot drive `restore_to`.
- *Render invalidation impact.* `CadProjection::tick` is head-advanced-driven (`crates/cad-projection/src/lib.rs:36–52`): when `CadGraph::head()` moves, all known entities are marked dirty next tick. Since World-only Actions never advance `CadGraph::head`, this approach does not change render-invalidation semantics. Selection-overlay rebuild (`rebuild_highlight_overlay` in `pick_path.rs`) is also not bus-driven today; it remains a direct call site.
- *Migration risk.* Lowest. No existing impls or tests need to change. Future commands that fit the contract (e.g. `set_time_scale`, future component edits) can be added incrementally. Risk is *content* rather than *contract*: many editor commands (CAD ops, selection, modal-state transitions) will be unable to use the bus at all, and so will accumulate parallel mutation paths.

**B. Widen `Action` to an editor command context.**

- *Dependency boundaries.* `editor-actions` would gain new mutation-target deps. To represent CAD-graph mutations (begin/commit/rollback against `CadGraph`) it would need a dep on `rge-cad-core`. To represent editor-state coordination mutations it would need a dep on `rge-editor-state`. Both are currently downstream of editor-actions in the dep graph (`editor-shell → {editor-actions, editor-state, cad-core, …}`); pulling `cad-core` and `editor-state` *into* `editor-actions` reshapes the crate's tier — it becomes a Tier-2 editor-domain coordinator instead of a thin kernel-adjacent mediation layer. The `forbidden-dep` architecture lint would need re-evaluation: `editor-actions` is not currently listed as a renderer or as `editor-ui`, so a tier-shift might not trip the existing rules directly, but it does invert the spirit of the doc comment ("the single mediation layer for editor mutations into runtime state").
- *Ownership authority.* The bus becomes the single mediator for all editor mutation surfaces: `World`, `CadGraph` (with checkpoint history), `editor-state`, and (optionally) projection/audit. Authority is concentrated; out-of-band mutations become illegal.
- *Audit / undo impact.* One unified undo timeline. Audit ledger and `CadGraph::CheckpointHistory` can co-cursor with the `UndoStack`. Eliminates the dual-ledger concern documented in `lifecycle/commands.rs:33–41`. But requires every existing impl (`SetTimeScale` and the test stubs in `editor-actions/src/{action.rs,bus.rs}`) to migrate to the widened signature; every test in `crates/editor-actions/tests/*.rs` and `crates/editor-shell/tests/{time_scale_test.rs,keyboard_command_bus_round_trip.rs,inspector_snapshot_smoke.rs}` would need re-shaping.
- *Render invalidation impact.* Improved: CAD-graph commits now travel through the bus, so the bus can be a witness for "head advanced" events and could (in a later dispatch) call `projection.tick` after each commit. Selection-overlay rebuild could similarly hook in deterministically.
- *Migration risk.* Highest. Touches a load-bearing trait, every existing impl, every test, and the editor-actions dependency graph. The crate's tier may need to change. Per `AGENTS.md` and the `editor-state-ownership` lint pattern documented in `editor-state/src/lib.rs`, expanding the mediation surface raises architectural sign-off questions that go beyond a single dispatch.

**C. Adapter layer around the existing bus.**

- *Dependency boundaries.* `editor-actions` keeps its current shape unchanged. New adapter types live in `editor-shell` (or a new `editor-shell::commands` submodule, alongside the existing `lifecycle/commands.rs`). Each "compound" editor command becomes a *wrapper function* that: captures pre-state on `CadGraph` / `editor-state` outside the `Action`; calls `EditorShell::submit_action` for the `World`-side effect; manually commits (or rolls back) the `CadGraph` operation; and records its own audit event on the editor-shell's `audit.rs` ring buffer (which is what `set_time_scale` already does in `lifecycle/commands.rs:381` via the dual-ledger pattern).
- *Ownership authority.* Bus authority is unchanged (`&mut World` only). The adapter holds composite authority — but only for one command at a time, at the call site. Different adapters can pick different compositions, which means semantics are per-adapter rather than per-trait. Risk: the bus's `is_dirty` / cursor only reflects the World mutation, not the CAD or coord side, so `Ctrl+Z` might leave the CAD graph or editor-state inconsistent unless the adapter author manually pairs an inverse CAD `restore_to` with each bus `undo`.
- *Audit / undo impact.* Dual (or triple) undo timelines persist. The bus's audit ledger records the World half of each compound command; the editor-shell `AuditEvent` ring records the editor-side half; `CadGraph::CheckpointHistory` records the CAD half. Replays must compose all three. The `set_time_scale` impl already demonstrates this dual-ledger pattern (`lifecycle/commands.rs:33–41`) and explicitly defers consolidation ("Ledger consolidation is a separate future dispatch").
- *Render invalidation impact.* Same as A unless the adapter pulls in `projection.tick` calls explicitly. Selection-overlay rebuild remains a direct call.
- *Migration risk.* Medium. No trait or `editor-actions` changes; no test rewrites. But every new compound command must hand-author its undo/audit pairing, which is precisely the divergence risk the original Command Bus design tried to eliminate (`bus.rs:76` — "the single mediation layer"). Long-term entropy is higher.

**Comparison summary.**

| Axis | A. World-only | B. Widened context | C. Adapter layer |
| --- | --- | --- | --- |
| Dep boundaries | unchanged | editor-actions gains cad-core + editor-state deps, tier shifts | unchanged |
| Ownership authority | bus owns World only | bus owns World + CadGraph + editor-state | bus owns World only; adapters own the rest |
| Audit / undo | one bus ledger + UndoStack; CAD + editor-state out-of-band | one unified timeline | dual/triple ledgers; per-adapter pairing |
| Render invalidation | unaffected; CAD head doesn't move | bus can drive `projection.tick` | same as A unless adapter calls it |
| Migration risk | lowest | highest (every impl + tests + dep graph + tier sign-off) | medium; divergence risk grows over time |

No recommendation is made in Q3 — Q5 names the smallest next step.

### Q4. Undo/Audit/Render Composition

For a future visible editor command (e.g. "click face to add it to selection", "drag transform gizmo to translate an entity", "extrude selected face by Δh") the composition needs to land in five places:

1. **`rge_kernel_ecs::World` mutation** (entity components / resources) — e.g., a `Transform` component edit, or a `TimeScale` resource update. *Already composes through the current bus.* `Action::apply(&mut World)` covers this directly; `SetTimeScale` demonstrates end-to-end (apply, revert via `insert_resource`, payload-driven coalesce).
2. **CAD operator-graph mutation** (`cad_core::CadGraph` via `begin_operation` → `graph_mut` → `commit`/`rollback`, or `restore_to`). *Does not compose through the current bus.* The bus has no handle to `EditorShell.cad_graph`; `Action::apply(&mut World)` cannot reach it. This requires either a widened context (approach B) or a hand-wired adapter (approach C). Without one of those, CAD commits have an independent undo timeline (`CheckpointHistory`), and `Ctrl+Z` against the bus reverts only the `World` half.
3. **Editor coordination state mutation** (`editor-state` — `Selection`, `Hover`, `ActiveTool`, `FaceSelection`, etc., owned at `EditorShell.coord`). *Does not compose through the current bus.* Same reason as (2): `coord` is not a `World`. `pick_path.rs::handle_left_click` mutates `self.coord.face_selection` directly today; nothing about that path knows of the bus. Same fix space as (2).
4. **Audit-ledger projection.** Two ledgers exist:
   - The bus's internal `rge_kernel_audit_ledger::AuditLedger` records `EventKind::Action` on each submit (`bus.rs:187`). *Already composes for every World-only submit.*
   - The editor-shell `audit::AuditLedger` (a ring buffer of `AuditEvent { PlayPressed, …, TimeScaleChanged, SnapshotCaptured, … }`) records lifecycle and editor-side events. *Does not compose through the bus*; each call site appends directly (`set_time_scale` does both: bus submit + `self.audit.record(AuditEvent::TimeScaleChanged{from,to})`). This dual-ledger is an explicit, locked design decision per `lifecycle/commands.rs:33–41`; consolidation is deferred.
5. **Snapshots, dirty state, render refresh.**
   - **`CommandBus::is_dirty()`** already drives the unsaved-changes flag (cursor-vs-save-mark on `UndoStack`); `mark_saved_command` clears it. *Composes for every World-only submit*; for editor-state-only or CAD-only mutations not routed through the bus, `is_dirty` does not flip — so the "any unsaved change" indicator may underreport unless those side mutations are bus-routed (approach B) or the adapter explicitly does a bus submit (approach C).
   - **PIE `WorldSnapshot`** captures and restores the kernel ECS world only (`crates/editor-shell/src/snapshot.rs`); editor-state intentionally persists across Play/Stop (per the `editor-state` doc-comment). Snapshots are orthogonal to the bus today and need no composition change for the bus integration itself.
   - **`CadProjection::tick`** is head-advance-driven (`cad-projection/src/lib.rs:36–52`). After a CAD-graph commit it re-projects dirty entities and refreshes `BRepHandle` payloads. **Composes correctly via the current loop** as long as `CadGraph::commit` happens at all; the open question is *who* makes the commit (bus, adapter, or direct call). Selection-overlay (`rebuild_highlight_overlay` in `pick_path.rs`) is direct-called from `handle_left_click`; if face-selection becomes bus-routed in the future, the overlay rebuild needs to fire from the bus-side (or from an adapter), not from the click handler.
   - **`InspectorSnapshot`** (`crates/editor-state/src/inspector_snapshot.rs`) is read-only observation. Composes regardless of approach.

**Already composes through the current bus.** World-only mutations (component / resource edits, `SetTimeScale`-style); bus's internal audit ledger; `UndoStack` cursor + save-mark; `is_dirty` for World-only mutations.

**Requires context design or adapter authority before it composes.** CAD-graph mutation (begin/commit/restore), editor-state coordination mutation, the editor-shell side audit ledger (whether to consolidate or keep dual-ledger), and the `is_dirty` flag's coverage of non-World mutations. Render invalidation composes structurally either way but needs a clear authority for *when* `CadProjection::tick` runs after a bus-routed CAD commit.

### Q5. Smallest Follow-Up

**NEEDS_HUMAN.**

The choice between the three approaches in Q3 is an architecture-tier decision that an executor cannot resolve from read-only code evidence alone. Specifically:

- Approach **A** (World-only) is technically immediate but *architecturally insufficient* for the load-bearing first input path identified in Q2 (left-click face selection mutates editor-state, not World) and for any future CAD-graph editor command (which must travel through `CadGraph::begin_operation`/`commit`).
- Approach **B** (widened editor context) requires promoting `editor-actions` from a kernel-adjacent mediation layer to a Tier-2 editor-domain coordinator (adding `rge-cad-core` and `rge-editor-state` to `crates/editor-actions/Cargo.toml`). This inverts the existing crate doc comment ("the single mediation layer for editor mutations into runtime state" — currently kernel-adjacent) and is a tier-graph reshape that affects the `forbidden-dep` lint posture across the workspace.
- Approach **C** (adapter layer) preserves the existing tier but accepts permanent dual-ledger + multiple undo timelines, contradicting the original "single mediation layer" intent and pushing per-command divergence risk onto every future adapter author.

The smallest **human** decision needed is: **pick the editor command-context shape — World-only Action, widened editor-context Action, or adapter-layer — with explicit sign-off on whether `editor-actions` may shift to a Tier-2 coordinator (B), or whether dual-ledger and parallel undo timelines are acceptable as a permanent posture (C), or whether the editor accepts that user actions touching CAD or editor-state will not be undoable via the bus (A).**
