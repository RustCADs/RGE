## Acceptance Criteria

- The Boolean fixture uses existing public APIs from `rge_cad_core` and
  `rge_cad_projection`. It must use `OperatorNode::Boolean(BooleanOp::...)` as
  the graph root, with two labeled Cuboid-derived inputs connected on Boolean
  ports 0 and 1.
- The fixture may follow the existing Boolean smoke pattern of using a
  translated second Cuboid to create an overlapping Boolean input, but the only
  source geometry must be Cuboid-derived and already labeled by the current
  Cuboid tessellation path. Do not synthesize face labels by hand.
- The test obtains the projected Boolean mesh through `projection.projected_mesh`
  and the render mesh through `projection.render_mesh_for`.
- `RenderMesh.face_labels` is asserted to be `Some`, non-empty, and the same
  length as both `projected.triangle_count()` and the projected mesh's
  `face_labels`.
- For every triangle index, the renderer-side `u64` label equals the projected
  `TopologyFaceId.0` value at that same triangle index. This must be checked in
  triangle order, not by sorting or set comparison.
- For every Boolean-root triangle index,
  `projection.brep_face_id_for_triangle(entity, tri, &world, graph.graph())`
  is asserted to return `None`.
- The test must not assert, require, or introduce any Boolean-root
  `BRepFaceId` mapping. Boolean remains topology-changing for B-Rep resolution
  in this dispatch.
- No production code, resolver arm, Boolean `BRepProvider`, renderer-crate
  dependency, Cargo dependency, or additional test file is introduced.
- `cargo test -p rge-cad-projection --test render_adapter_smoke` exits 0.

## Planner Notes

`crates/cad-projection/tests/render_adapter_smoke.rs` already contains the
per-operator render-adapter smokes for Cuboid, Extrude, RoundFillet, Fillet,
Transform, Sweep, Loft, and Revolve. Add Boolean as the next operator-specific
case in that same file to keep the existing cohesive smoke aggregation intact.

Boolean differs from the recent Sweep, Loft, and Revolve cases: its rendered
labels should still mirror the projected `TopologyFaceId` values in order, but
the graph-level B-Rep resolver intentionally treats Boolean as
topology-changing. The correct assertion is therefore `None` from
`brep_face_id_for_triangle` for Boolean-root triangles, not a fabricated or
inherited `BRepFaceId`.
