## Goal

Implement GitHub issue #196 by adding a narrowly-scoped one-shot transient recovery requeue path to `Invoke-AiDispatchAuto.ps1`. The autonomous driver currently halts on any `ai-auto` issue carrying `ai-dispatch-failed`; this dispatch permits it to recover exactly one open autonomous failure when the failure taxonomy is clearly transient (`ai-dispatch-failure-stall` or `ai-dispatch-failure-timeout`) and the issue has not already been recovered, while preserving every existing hard stop for local sentinels, non-transient failures, ambiguous failure sets, and human-review states.

## Deliverables

- A change in `Invoke-AiDispatchAuto.ps1` that keeps `.ai/dispatch.auto-halt` as the first halt check before any GitHub issue recovery query or label mutation.
- A one-shot recovery decision path after the sentinel check and before the existing failed-auto halt, limited to open issues labelled both `ai-auto` and `ai-dispatch-failed`.
- Recovery eligibility that is true only when exactly one open failed autonomous issue exists, it has exactly one transient taxonomy label from `ai-dispatch-failure-stall` or `ai-dispatch-failure-timeout`, it does not have `ai-dispatch-recovered-transient`, and it has no non-transient failure taxonomy labels.
- Recovery in normal mode that requeues the eligible issue by removing `ai-dispatch-failed`, removing `ai-dispatch-done` if present, and adding `ai-dispatch`, `ai-dispatch-retry`, and `ai-dispatch-recovered-transient` while keeping the existing transient taxonomy label on the issue.

- Durable fail-closed output for ineligible states: closed failures, multiple simultaneous failed issues, already-recovered issues, mixed taxonomy labels, `ai-dispatch-failure-blocked`, `ai-dispatch-failure-verification`, `ai-dispatch-failure-control`, `ai-dispatch-failure-publish`, `ai-dispatch-failure-unknown`, missing taxonomy labels, or unknown future `ai-dispatch-failure-*` labels.

## Acceptance Criteria

- The first halt condition in `Invoke-AiDispatchAuto.ps1` remains the local sentinel check for `.ai/dispatch.auto-halt`; no `gh issue list`, `gh issue view`, `gh issue edit`, recovery helper, or other GitHub recovery logic runs before that sentinel check.
- The script still exits 0 without touching GitHub labels when `.ai/dispatch.auto-halt` exists.
- The failed-auto recovery query considers only open issues labelled `ai-auto` and `ai-dispatch-failed`; closed issues must not be recovered and must not force transient recovery.
- If exactly one open failed autonomous issue is eligible, normal mode performs one GitHub label mutation equivalent to:
  - remove `ai-dispatch-failed`
  - remove `ai-dispatch-done` if present
  - add `ai-dispatch`
  - add `ai-dispatch-retry`
  - add `ai-dispatch-recovered-transient`
  - leave `ai-dispatch-failure-stall` or `ai-dispatch-failure-timeout` in place
- If the eligible issue is recovered, the script prints a clear recovery message and then continues toward the normal queue-drain path for the requeued issue rather than selecting a new task in the same tick.
- `-DryRun` prints whether recovery would occur and, when eligible, prints the issue number plus labels to remove/add, but it must not call `gh issue edit`, create issues, or run `Invoke-AiDispatchQueue.ps1`.
- Already recovered issues carrying `ai-dispatch-recovered-transient` are never recovered again; they continue to halt for human review.
- Multiple simultaneous open issues labelled both `ai-auto` and `ai-dispatch-failed` are never recovered automatically, even if all look transient.
- Issues with no taxonomy label, more than one taxonomy label, or any taxonomy label outside the two transient labels are never recovered automatically.
- The recovery path does not change queue retry policy, publish behavior, issue close behavior, branch handling, commit staging, queue scope guard behavior, task selection prompt semantics, JSONL timing trace schema, or label taxonomy classification.
- The implementation remains compatible with Windows PowerShell 5.1, including single-item array handling for `ConvertFrom-Json` results.
- The only tracked source change is `Invoke-AiDispatchAuto.ps1`; generated or untracked changes are limited to this dispatch's own `ai_handoffs/ISSUE-196_EXEC_*.md` report and queue-created `ai_dispatch_logs/log_*.md` artifact.

## Planner Notes

This task intentionally puts recovery in the Auto layer, not in Queue. Queue remains responsible for classifying terminal failures and deciding whether the first failed attempt gets its built-in retry. Auto only gets a narrow repair hook for a later scheduler tick that sees a terminal failed autonomous issue whose taxonomy indicates a transient infrastructure condition.

The intended ordering is strict: local `.ai/dispatch.auto-halt` sentinel first, then the open failed autonomous issue check and optional one-shot transient recovery, then the existing human-review halt behavior for anything unrecovered. This preserves the durable local stop switch even if GitHub contains a recoverable-looking transient issue.

Treat `ai-dispatch-recovered-transient` as the one-shot guard. It should be created idempotently if needed before use, with a concise description, but it must never be removed by this recovery path. Keeping the original transient taxonomy label is useful audit evidence; removing only `ai-dispatch-failed` and optional `ai-dispatch-done` is enough to stop the halt and requeue the issue.

Under the orchestrator protocol, no `_REVIEW_` or `_CLOSEOUT_` packet is expected from this TASK. The Executor should produce one `ISSUE-196_EXEC_*.md` report for this run.
