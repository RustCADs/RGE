## Per-File Summary

`Invoke-AiDispatchQueue.ps1` (+251 / −0):

- Introduced `Invoke-QueueScopeGuard`, called immediately after
  `Write-DispatchLog` returns the dispatch log path and before
  `Git-Step @('add', '-A')`. The guard reads
  `git -c core.quotepath=false status --short --untracked-files=all`,
  classifies every changed/deleted/renamed/copied/untracked path against the
  active dispatch's allowed surface, and aborts via `Fail` (which releases
  the queue lock and exits non-zero) if any disallowed path is present.
- Allowed surfaces, in evaluation order per path:
  1. `Test-ActiveDispatchArtifactPath` — anchored regex
     `^ai_handoffs/([^/]+)$` enforces a single basename directly under
     `ai_handoffs/`; the basename must match
     `^<DispatchId>_(TASK|EXEC|CORRECT)_[^/]+\.(md|meta\.json)$`. Nested
     paths like `ai_handoffs/ISSUE-180_EXEC_x/nested.md` and REVIEW/CLOSEOUT
     basenames are rejected.
  2. `Test-ExactQueueLogPath` — admits exactly the `Write-DispatchLog`
     return path, and only when that path is directly under
     `ai_dispatch_logs/` matching `log_*.md`. No TASK token can broaden it.
  3. `Test-TaskTokenMatchesPath` against TASK-derived positive tokens.
- `Get-TaskPositiveAllowedTokens` walks the active TASK packet line by line,
  toggling on `### MAY edit` / `### MAY add new files` headings and off on
  any other heading, while skipping fenced code blocks so a `#` inside a
  sample does not look like a section heading. It harvests backtick-quoted
  tokens, passes them through `Test-LooksLikePathToken` (require a `/`, a
  `*`, or a `.<ext>` tail so bare identifiers like `Write-DispatchLog` or
  `git add` are dropped), normalizes to repo-relative forward-slash form,
  and quarantines tokens under `ai_handoffs/` or `ai_dispatch_logs/` so
  those trees remain governed solely by the hard-coded artifact rules.
- `Convert-GlobToRegexForQueueGuard` keeps `*` segment-bounded (`[^/]*`)
  while letting `**` cross separators (`.*`); `?` is `[^/]`; every other
  regex metachar is escaped. This stops `foo/*.md` from accidentally
  matching `foo/sub/x.md`.
- Non-glob tokens go through a path-boundary check: `path -eq token` OR
  `path.StartsWith(token + "/")`, so a TASK directory token `some/dir`
  allows `some/dir/file.txt` but never `some/dir2/file.txt`.
- `Get-QueueStatusEntries` parses the porcelain output into entries that
  carry one path for normal statuses and TWO paths (old + new) for
  rename/copy statuses (`R*` / `C*`). The guard checks both paths of a
  rename/copy individually, so a rename that moves work into an allowed
  destination still fails if the source path was not part of the allowlist.
- `Get-ActiveTaskPacketPathForQueueGuard` finds the newest TASK packet for
  the active dispatch under `ai_handoffs/` by lexicographic basename sort.
  If none exists, the guard fails closed with a clear message; if the
  packet exists but yields zero positive path tokens, the guard also fails
  closed before any staging.
- On disallowed paths the guard emits a single multi-line `Fail` message
  that names every disallowed path with its git status code, plus the
  active TASK packet and dispatch log paths for context. Because `Fail`
  releases the lock and exits non-zero, no `git add`, commit, merge, push,
  publish, or branch deletion runs for the failing run.

## Open Questions for Reviewer

- The TASK-token parser deliberately filters backtick contents through a
  `Test-LooksLikePathToken` heuristic (must contain `/`, `*`, or end with a
  `.<1..8-char ext>`) to keep identifiers like `Write-DispatchLog` and
  arguments like `--repo` out of the allowlist. Please confirm that
  heuristic is acceptable, or call out a TASK shape it would mis-classify.
- The guard treats a non-glob token as both an exact file match and a
  path-boundary directory prefix (`path -eq tok` OR `path.StartsWith(tok +
  "/")`). For tokens that name a specific file (e.g.,
  `Invoke-AiDispatchQueue.ps1`) the prefix branch is a no-op because the
  filesystem cannot host both a file and a directory at the same path; the
  combined check just keeps the rule uniform for directory-prefix tokens.
- Section parsing keys on heading text `### MAY edit` and
  `### MAY add new files` (case-sensitive on `MAY`). If a future TASK
  reformats those headings (e.g., `### May edit`, `### MAY EDIT`,
  `## MAY edit`) the guard will see zero positive tokens and fail closed —
  intentional, but worth flagging if the planner's heading conventions are
  expected to drift.
