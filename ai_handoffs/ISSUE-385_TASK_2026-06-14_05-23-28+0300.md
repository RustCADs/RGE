## Deliverables

- A host-local main-menu item annotation path that exposes deterministic conflict peer detail for enabled menu entries whose displayed shortcut exactly matches one `ProjectedMainMenu.conflicts` shortcut.
- Conflict detail copied from the matching `ProjectedShortcutConflict.entries` vector, preserving projection order and without synthesizing or re-resolving peer ids from registries.
- Unconflicted enabled main-menu entries expose no conflict detail.
- Disabled main-menu entries keep existing disabled behavior and do not gain conflict detail even when their displayed shortcut appears in `ProjectedMainMenu.conflicts`.
- Projected plugin menu entries can expose conflict detail when the existing projected menu data says they are enabled and conflicted; no plugin runtime/discovery/loading or registry re-resolution is added.
- Focused `menu_tests.rs` coverage for enabled conflicted, unconflicted enabled, disabled-conflicted, and plugin-projected main-menu annotation behavior, plus unchanged ordering/shortcut text/command enqueueing where relevant.
- Status/task docs mark task 131 complete and do not append task 132.

## Acceptance Criteria

- Main-menu conflict matching uses only the already-projected `main_menu.conflicts` data and the item's displayed shortcut string.
- Conflict detail is attached only when the menu entry is enabled and the displayed shortcut exactly matches a `ProjectedShortcutConflict.shortcut`.
- Peer ids are copied from `ProjectedShortcutConflict.entries` preserving their existing order.
- The implementation does not append conflict text to the shortcut display string, replace the menu label, reorder menu entries, or change which `Command` is enqueued for any clicked menu item.
- The existing menu item click response remains the source of command enqueueing; added conflict detail must not create a second command target, make disabled items activatable, or change top-level menu open/close behavior.
- Command-palette projection, command-palette activation, shortcut-help rows, Shortcut Conflicts rows, shortcut execution, `MenuCommandHandoff`, and `EditorShell::route_menu_command` behavior remain unchanged.
- `crates/editor-egui-host/src/lib.rs` is edited only if the existing main-menu call sites need to pass already-projected conflict detail into `menu_item`.
- `.ai/dispatch.tasks.md` has no new `132.` heading after implementation.
