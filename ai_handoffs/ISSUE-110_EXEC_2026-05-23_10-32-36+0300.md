## IO Formats Metadata Preflight Answers

### 1. What metadata shape does the lint read?

The `kernel-isolation` lint at `tools/architecture-lints/src/kernel_isolation.rs`
reads each workspace package's Cargo metadata via `cargo metadata`, identifies
the `io-*` packages, and then extracts the `package.metadata.rge.formats` array
from each. Specifically, `extract_formats` walks
`pkg.metadata.get("rge").and_then(|rge| rge.get("formats")).and_then(|f| f.as_array())`
and lower-cases every string entry. Concretely (see lines 19-35 and 73-87):

```toml
# crates/io-<name>/Cargo.toml
[package.metadata.rge]
formats = ["gltf", "glb"]
```

Rules verified from the source and from the integration tests in
`tools/architecture-lints/tests/kernel_isolation_test.rs`:

- Each string is an extension name, lower-case, **no leading dot**
  (see crate doc-comment lines 27-28 and `extract_formats` calls
  `str::to_lowercase` on every entry).
- A format claimed by ≥ 2 detected `io-*` crates is a violation (exit 1)
  (lines 128-157; integration tests 4, 5, 6, 8 cover this).
- Missing `[package.metadata.rge]` on an `io-*` crate is **only a stderr
  warning**, never a violation (Option B, lines 30-35 of the source and
  the `test_missing_metadata_is_warning_not_violation` test, lines
  301-334 of the test file). The lint still exits 0.
- No alias normalization is performed inside the lint: `jpg` and `jpeg`
  are treated as distinct extension strings; the lint only compares
  lower-cased literals. Avoiding overlap-by-aliasing therefore requires
  manifest discipline, not lint logic — see resolution in §4.

### 4. Which ownership ambiguities and aliases are resolved, and which remain?

Resolved (the implied four arrays above already encode these):

- **`jpg` and `jpeg`** — both belong to `rge-io-image`. The lint
  string-compares lower-cased literals and performs no alias collapse
  (`format_detect.rs` and `extract_formats` carry no normalization),
  so both spellings must be claimed by io-image. Cargo description and
  feature flag `image = { ..., features = ["png", "jpeg", "hdr"] }`
  (manifest line 13) confirm JPEG is owned here.
- **`step` and `stp`** — both belong to `rge-io-step`. Cargo description
  "STEP/IGES import via OCCT" covers both spellings of the STEP family.
- **`iges` and `igs`** — both belong to `rge-io-step`. Same Cargo
  description covers both spellings of the IGES family. The combined
  array `["step", "stp", "iges", "igs"]` therefore consolidates STEP
  and IGES ownership under one crate, matching the manifest's stated
  scope.
- **`gltf` and `glb`** — both belong to `rge-io-gltf`. The `gltf`
  workspace dependency is gated to `default-features = false` with
  only `["utils", "names"]` features (manifest lines 16-18) precisely
  to keep this crate the sole owner of the glTF container (per the
  one-importer-per-format rule, PLAN §1.6.4/§1.6.5 referenced in the
  crate's own `lib.rs` and `Cargo.toml` comments).
- **Embedded raster bytes inside glTF / GLB** — codec-owned by
  `rge-io-image`. `crates/io-gltf/src/image.rs:1-44` is explicit:
  io-gltf extracts buffer-view bytes and `data:image/...;base64,...`
  URIs from the glTF document, then **delegates decoding** to
  `rge_io_image::load_bytes`. The crate-level doc on `image.rs` and
  the dependency-rationale comment in `crates/io-gltf/Cargo.toml`
  lines 12-29 spell this out: "io-image stays the single owner of
  raster codecs; io-gltf only handles the glTF embedding semantics."
  Therefore raster extension strings (`png`, `jpg`, `jpeg`, etc.) must
  NOT appear in `rge-io-gltf`'s formats array — they belong to io-image
  and adding them to io-gltf would trigger the overlap violation.

Remaining (intentionally not resolved here):

- Ownership of formats for `io-obj`, `io-audio`, and `io-3mf` is out of
  scope per the Task Packet's Constraints / Non-Goals. The Task Packet
  explicitly forbids deciding their arrays in this dispatch; they are
  listed in §2 as detected workspace IO members only.
