### Q2. Input Routing

**Smallest safe input-routing shape (descriptive only — the existing
wiring already implements this shape):**

- **Platform-input layer** = `editor-shell`. `editor-shell` owns the
  winit `ApplicationHandler` and receives raw `WindowEvent`s in
  `lifecycle::window_event`. Per audit, this is the only crate in the
  audit roots that consumes raw winit events; the host crate must NOT
  re-implement `ApplicationHandler` (would cycle into editor-shell).
- **Egui-consumption decision layer** = the host crate, exposed as a
  single `on_window_event(&mut self, window, event) -> EventResponse`
  call. The host wraps `egui_winit::State::on_window_event`, which is
  the canonical adapter that returns `{ consumed: bool, repaint: bool
  }` for a given `WindowEvent`. Per audit, `editor-shell/src/
  lifecycle/mod.rs:1544-1545` already calls this adapter as
  `host.on_window_event(window, &event)` BEFORE any picker / bus
  branch and stores the resulting `consumed` flag.
- **Pre-existing input boundaries that MUST remain bounded:**
  - The CommandBus + UndoStack pipeline in `editor-actions` continues
    to receive only `EditorKeyCommand` translations of winit
    `KeyboardInput` events — gated on `!egui_consumed` so a key typed
    into an egui text field is not double-applied as a bus command.
    Audit citation: `lifecycle/mod.rs:1688-1700`.
  - The face-picker in `editor-shell::pick_path` continues to receive
    only `MouseInput` events that satisfy
    `should_fire_face_pick(egui_consumed, over_viewport_tab) =
    !egui_consumed || over_viewport_tab`. The exception (egui-consumed
    but pointer over the viewport tab body) is documented at
    `lifecycle/mod.rs:1457-1483`; it is load-bearing because the
    egui_dock area covers the whole window, so without it every
    viewport click would be eaten by egui.
  - The `EditorKeyCommand` translation continues to live in
    `crates/input::translate_keyboard`; editor-shell owns the only
    `rge-input` dep (rule 4 still forbids editor-ui from doing so).
    Adding an editor-egui-host → input edge is not required and is
    not present in the audit; the host does not need keyboard
    translation.
- **`ModifiersChanged` + `CursorMoved` must reach both the host and
  the editor's own modifier-state machine** (audit citation:
  `lifecycle/mod.rs:1540` — "ModifiersChanged + CursorMoved are
  observed by BOTH egui and..."). This is not a single-consumer event;
  the host gets a copy via `on_window_event` and the picker / bus keep
  their own copy of the current modifier mask.
- **Resize + scale-factor changes are forwarded to the host
  explicitly, not via `on_window_event`** (audit citation:
  `lifecycle/mod.rs:1597-1602`). The host needs the new logical size +
  dpi to reconfigure its `egui_wgpu::Renderer` viewport state and its
  `egui::Context::set_pixels_per_point`. This is a separate API
  surface (`host.on_resized` / `host.on_scale_factor_changed`)
  deliberately distinct from `on_window_event` because resize is also
  consumed by the wgpu surface reconfigure path in
  `editor-shell::render_path`.

### Q3. Render Composition

**Design-level render composition (descriptive — the existing wiring
already implements this composition):**

- **Egui frame lifecycle**:
  1. `egui::Context::begin_frame` (host) at the start of the egui
     pass, consuming the `RawInput` accumulated by `egui-winit::State`
     since the previous frame.
  2. User-supplied closure runs all UI code against `&mut egui::Ui`
     (`DockArea::show`, widget calls, etc.).
  3. `egui::Context::end_frame` returns a `FullOutput` containing the
     `ShapeVec` + texture-delta + platform-output.
  4. `egui_wgpu::Renderer::update_buffers` uploads vertex / index /
     uniform buffers for the frame.
  5. `egui_wgpu::Renderer::render` records a render pass that paints
     the ShapeVec into the target view.
- **Draw-data / texture handling**: textures are owned by
  `egui_wgpu::Renderer` (which maintains a `HashMap<TextureId,
  wgpu::Texture>` and applies `TexturesDelta::set` / `::free` each
  frame). The host crate keeps this entirely internal; the editor
  never touches an egui texture directly.
- **Relationship to the existing render path**: the existing main
  pass in `editor-shell::render_path` clears the surface, sets the
  lit-mesh pipeline, encodes one cuboid `draw_indexed`, optionally
  encodes a highlight overlay, and closes the pass. The egui pass is
  recorded **into the same `wgpu::CommandEncoder` immediately after
  the highlight overlay close and before `queue().submit()`**, using
  `LoadOp::Load` so the cuboid + highlight pixels are preserved and
  egui paints on top. This avoids a second `submit()` and keeps the
  frame as a single GPU command stream. Audit citation:
  `editor-shell/src/render_path.rs:366-418` and the in-source comment
  at `:843-878` noting the visual-test harness deliberately excludes
  the egui pass.
- **Surfaces that MUST remain outside any follow-up dispatch**:
  - The cuboid + highlight passes (no edits to lit-mesh pipeline,
    camera bind groups, material bind groups).
  - The visual-test harness in
    `editor-shell/src/visual_test_harness.rs` — it deliberately
    renders egui-free for byte-stable pixel goldens, so its
    `encode_main_pass` path must not pick up an unconditional egui
    pass.
  - `editor-shell::render_input::RenderHandoff` (the existing
    sim → render boundary). The egui pass consumes UI-side state, not
    sim-side state; it does not need a RenderHandoff slot.
  - Any wgpu surface acquisition / configuration — those stay in
    `editor-shell::render_path`. The host receives a `wgpu::
    TextureView` + `wgpu::CommandEncoder` from the caller; it never
    owns the surface.
  - Inspector-tab body wiring (per planner notes; that is a
    deliberately separate later dispatch).

### Q4. DockState/Snapshot Ownership

**Ownership rules (descriptive — already realised by the existing
wiring):**

- **`DockState<TabBody>`** is owned by the host crate
  (`crates/editor-egui-host`), not by `editor-shell` and not by
  `editor-ui`. Rationale:
  - `editor-ui` defines `TabBody` shapes + widget render functions
    but must remain headless (no wgpu, no winit, no egui Context);
    putting `DockState` on an editor-ui type would force editor-ui
    to take a live host borrow per frame.
  - `editor-shell` is the platform-input + render-path skeleton and
