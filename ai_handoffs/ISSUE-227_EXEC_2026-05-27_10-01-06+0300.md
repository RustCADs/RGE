## What I Changed

### Source

- `Invoke-AiDispatchQueue.ps1` (after `Get-FailureTaxonomyLabels`,
  ~line 1049): added the pure helper `Get-DispatchTerminalLabelPlan`.
  It accepts the already-computed queue state (`WillRetry`,
  `RunFailed`, `QueueLabel`, `RunLabel`, `DoneLabel`, `FailLabel`,
  `RetryLabel`, `TaxonomyLabels`, `KnownFailureTaxonomyLabels`) and
  returns `[pscustomobject]@{ Add = @(...); Remove = @(...) }` covering:
  - Terminal success: Add = `{done}`; Remove = `{queue, running, retry,
    failed, *every known taxonomy label*}`.
  - Terminal failure: Add = `{done, failed, *selected taxonomy*}`;
    Remove = `{queue, running, retry, *non-selected taxonomy labels
    from the known set*}`.
  - Retry: Add = `{queue, retry}`; Remove = `{running, done, failed,
    *every known taxonomy label*}`.
  The helper deduplicates while preserving first-occurrence order, never
  reads/writes files, never calls `gh`/`git`/`codex`/`claude`/network,
  and is decorated `[CmdletBinding()]`.
- `Invoke-AiDispatchQueue.ps1` (terminal relabel block, ~line 1543):
  derived `$knownFailureTaxonomyLabels` once by filtering `$labelSpec`
  for `ai-dispatch-failure-*` (so the label list and label-spec stay in
  sync from a single source) and replaced the per-branch `--add-label`/
  `--remove-label` construction with a `foreach` loop over the helper's
  `Add` / `Remove` arrays.
- `Invoke-AiDispatchQueue.ps1` (verification block, ~line 1561):
  rewrote the `$labelOk` check to walk `$labelPlan.Add` (assert
  `-contains`) and `$labelPlan.Remove` (assert `-notcontains`) uniformly
  across all three states, so incompatible-label survivors now fail the
  finalization just like missing-required-label cases already did.

### Tests

- `tools/dispatch-tests/Get-DispatchTerminalLabelPlan.Tests.ps1`: new
  Pester file with 15 `It` blocks organized under six `Context` blocks.
  The tests dot-source `Invoke-AiDispatchQueue.ps1` via the existing
  `RGE_AI_DISPATCH_QUEUE_SKIP_MAIN=1` testability seam so only the
  helper definitions load — no live `gh`/`codex`/`claude`/scheduler/
  queue/network invocation, no real GitHub issue creation, no temp git
  repo. Coverage:
  - Helper is exposed after dot-sourcing.
  - Terminal success — add-set is exactly `{done}`, remove-set covers
    queue/running/retry/failed plus every known taxonomy label, and
    `done` never appears in remove.
  - Terminal failure (single taxonomy `ai-dispatch-failure-control`) —
    add-set has `{done, failed, control}`; non-selected taxonomy
    labels are all removed; `failed`/`done`/selected taxonomy never
    appear in remove.
  - Terminal failure (multi taxonomy `verification` + `publish`) —
    add/remove partition cleanly.
  - Retry — add-set has `{queue, retry}`; remove-set has
    running/done/failed plus every known taxonomy label (including any
    accidentally selected one); queue/retry never appear in remove.
  - Stale/incompatible removal — passing run scrubs every known
    taxonomy label; classification switch (`timeout` → `verification`)
    plans the stale `timeout` removal alongside the new add.
  - Add/Remove non-overlap across all five representative states.
  - Determinism (two consecutive calls return identical add/remove
    ordering) and dedup (duplicate selected taxonomy collapses to a
    single add).

## Open Questions for Reviewer

- The known failure-taxonomy label set is derived from `$labelSpec`
  via `Where-Object { $_.Name -like 'ai-dispatch-failure-*' }`. This
  keeps a single source of truth (`$labelSpec`) and avoids drifting a
  separate parallel array, but it does couple the helper's "scrub all
  taxonomy labels" behavior to whatever future entries the label spec
  acquires. If a future failure label is added to `$labelSpec` without
  the `ai-dispatch-failure-` prefix, the new label will silently
  escape the scrub. Worth a callout in the §14.13 entry, or is this
  acceptable as the documented prefix convention?
