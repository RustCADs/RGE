## Goal

Add an opt-in `pr` publish mode for GitHub issue #230 so a successful,
control-passed AI dispatch pushes its `ai-dispatch/ISSUE-number` branch and
opens a GitHub pull request targeting `main`, while preserving the existing
`main` auto-publish default and the existing `-NoPublish` branch-only behavior.
PR mode must automate branch push plus PR creation, but must not fast-forward,
merge, push `origin/main`, or close the source issue.

## Deliverables

- A queue-level publish-mode interface in `Invoke-AiDispatchQueue.ps1` that
  supports `main`, `branch`, and `pr`.
- Backward-compatible `-NoPublish` handling that continues to mean branch-only
  mode. If `-NoPublish` is combined with an explicit non-branch publish mode,
  fail fast with a clear parameter error rather than guessing.
- Preserved queue default behavior: callers that do not specify publish mode
  and do not pass `-NoPublish` still use `main` auto-publish mode.
- A PR publish path that runs only after a branch commit exists, the dispatch
  loop exits 0, and Codex control verdict is `pass`.
- PR mode branch publication that pushes `ai-dispatch/ISSUE-number` to
  `origin` and creates a pull request targeting `main`.
- Deterministic PR title/body generation, preferably via a pure helper that is
  easy to test without live GitHub calls.
- PR body content that includes the source issue number, dispatch id, branch,
  commit SHA, detailed dispatch-log path, Codex control verdict, and the exact
  text `Refs #230` for this issue shape. The implementation should generalize
  the issue number for future issues and must not use `Closes #...`.
- Final result issue comments that include the PR number and URL when PR
  creation succeeds.
- Publish failure handling in PR mode that classifies branch-push or PR-create
  failure as `ai-dispatch-failure-publish`, leaves the local branch available
  for human recovery, avoids retrying it as an accidental execution failure,
  and never falls back to pushing `origin/main`.
- `Invoke-AiDispatchAuto.ps1 -PublishMode pr` support that routes to the
  queue's PR mode.
- `Register-AiDispatchSchedule.ps1 -Autonomous -PublishMode pr` support that
  records scheduled arguments with `-PublishMode pr`.
- Updated mid-run progress comment text so the `publish-decision` stage
  distinguishes PR mode from `main`, `branch`, `not-eligible`, and `no-commit`.
- Updated `AI_DISPATCH_AUTOMATION.md` documentation describing `branch`,
  `main`, and `pr` publish modes, including the fact that PR mode pushes the
  dispatch branch and opens a PR but does not merge, push `origin/main`, or
  close the source issue.

## Acceptance Criteria

- `Invoke-AiDispatchQueue.ps1` accepts `-PublishMode main`, `-PublishMode branch`,
  and `-PublishMode pr`.
- Existing `Invoke-AiDispatchQueue.ps1 -NoPublish` remains accepted and behaves
  as branch mode.
- Existing queue default behavior remains main auto-publish mode unless the
  caller explicitly requests another mode.
- `Invoke-AiDispatchAuto.ps1 -PublishMode pr` is accepted and invokes the queue
  in PR mode.
- `Register-AiDispatchSchedule.ps1 -Autonomous -PublishMode pr` is accepted and
  its registered action arguments include `-PublishMode pr`.
- PR mode success pushes the dispatch branch to `origin` and opens a pull
  request targeting `main`.
- PR mode never runs `git merge --ff-only`, never pushes `origin main`, never
  pushes `origin/main`, and never closes the source issue automatically.
- PR text uses `Refs #issue-number` and never `Closes #issue-number`.
- The final issue result comment includes the PR number and URL on successful
  PR creation.
- Branch-push or PR-create failure in PR mode is treated as a publish-pipeline
  failure with `ai-dispatch-failure-publish`, with no automatic retry and with
  the local dispatch branch preserved for human recovery.
- Progress comments for the `publish-decision` stage include a distinct PR-mode
  decision string.
- Focused pure-helper or Pester tests cover publish-mode normalization, PR body
  generation, or both without live GitHub, network, branch push, PR creation,
  or issue mutation calls.
- The canonical verification gate passes.

## Planner Notes

Current behavior uses `-NoPublish` as the queue's branch-only switch and treats
absence of that switch as main auto-publish. The least disruptive shape is a
small normalization layer in `Invoke-AiDispatchQueue.ps1` that resolves
`-PublishMode` plus `-NoPublish` into one internal mode before progress comments,
publish decisions, result comments, retry classification, issue close, and final
reporting are computed.

Prefer isolating deterministic PR title/body formatting and publish-mode
normalization into pure helpers so `tools/dispatch-tests/**` can verify the new
behavior without invoking `gh`, `git push`, or the queue main flow. Existing
tests already dot-source the queue through `RGE_AI_DISPATCH_QUEUE_SKIP_MAIN`;
reuse that pattern rather than adding a new test harness.

The intended PR publish command shape is branch push to `origin` followed by
`gh pr create --base main --head ai-dispatch/ISSUE-number --title ... --body-file ...`
or an equivalent `gh` invocation. Capture the resulting PR URL and, if needed,
query `gh pr view` for the number. Any failure in branch push, PR create, or PR
metadata lookup should be publish-pipeline failure, not an execution retry.
