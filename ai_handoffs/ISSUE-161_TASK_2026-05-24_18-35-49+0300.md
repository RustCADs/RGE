## Planner Notes

ISSUE-158 established that raw `ComponentValue.data` payloads are schema-only
safe in `rge-data`, but it routed canonical type-id ambiguity to human input.
GitHub issue #161 supplies that decision: the fixture must use these exact
scene-envelope type-id strings:

```text
rge::components::Transform
rge::components::Camera
rge::components::Light
rge::components::Visibility
```

Keep the implementation intentionally narrow. The fixture should store raw RON
strings; the test should parse those strings with `ron::Value` to prove the
shape is valid RON and assert the expected field/variant facts. Do not import
or depend on the actual component crates from `rge-data`.

The intended raw payloads are:

```text
Transform: (translation:(0.0,0.0,5.0),rotation:(0.0,0.0,0.0,1.0),scale:(1.0,1.0,1.0))
Camera: (projection: Perspective(fov_y_radians: 1.0471976, near: 0.05, far: 1000.0), viewport: (0.0, 0.0, 1.0, 1.0), priority: 0, is_active: true)
Light: (color:(1.0,1.0,1.0),kind:Directional(illuminance_lux:100000.0),affects_indirect:true)
Visibility: Visible
```

The literal `fov_y_radians: 1.0471976` must be present in the fixture and the
test expectation. The test should also inspect the parsed `ron::Value` enough
to prove that value belongs to the `fov_y_radians` field of the `Perspective`
projection, not merely a substring elsewhere in the payload.
