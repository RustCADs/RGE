## Goal

Harden `Invoke-AiDispatchQueue.ps1` so the queue cannot accidentally stage,
commit, or publish changed or untracked files outside the active dispatch's
declared task surface. The queue must validate the worktree after writing its
own dispatch log and before broad branch staging, allowing only active-dispatch
handoff artifacts, the exact queue log it just wrote, and positive allowed paths
parsed from the active TASK packet.

## Deliverables

- Add a queue staging guard in `Invoke-AiDispatchQueue.ps1` that runs after
  `Write-DispatchLog` returns the queue log path and before any broad
  `git add -A`, commit, checkout-to-main, merge, push, or publish step.
- The guard must inspect all changed, deleted, renamed, copied, and untracked
  repo-relative paths that `git status --short --untracked-files=all` reports
  before staging.
- The guard must allow only these surfaces:
  - Active-dispatch handoff artifacts with a single basename directly under
    `ai_handoffs/`: `ISSUE-180_TASK_2026-05-25_18-55-31+0300.md`,
    `ISSUE-180_EXEC_2026-05-25_19-00-00+0300.md`, and
    `ISSUE-180_CORRECT_2026-05-25_19-00-00+0300.md` illustrate the required
    basename shape for the active dispatch.
  - Matching `.meta.json` sidecars for those same active-dispatch TASK, EXEC,
    and CORRECT basenames.
  - The exact queue log path returned by `Write-DispatchLog`, provided it is
    directly under `ai_dispatch_logs/` and its basename matches `log_*.md`.
  - Positive allowed paths parsed from the active TASK packet's positive
    allowed-path sections.
- Positive allowed-path parsing must use only the active TASK packet's
  `### MAY edit` and `### MAY add new files` sections. It must not parse
  `RELATED_FILES`, deliverables, acceptance criteria, constraints, planner
  notes, prior feedback, halt conditions, `MUST NOT edit`, `MUST NOT add new
  files`, or any other negative or forbidden section.
- Positive TASK path parsing must preserve both glob patterns and directory
  prefixes. Bare affirmative directory-prefix tokens such as `some/dir`,
  `some/dir/`, and `some/dir/**` must allow files below that directory with a
  path-boundary match, without allowing sibling paths such as `some/dir2/file`.
- Special artifact/log allowances must not be broadened by TASK parsing.
  Ignore or quarantine positive TASK tokens under `ai_handoffs/**` and
  `ai_dispatch_logs/**`; those trees are allowed only by the hard-coded
  active-dispatch artifact rules and the exact just-written dispatch log path.
- If the guard finds any disallowed changed or untracked path, it must print a
  clear list of every disallowed path and fail before staging, committing,
  merging, pushing, publishing, or deleting the dispatch branch.
- If no positive allowed file surface can be parsed from the active TASK packet,
  the guard must fail closed before staging.
- Preserve existing queue behavior when every changed or untracked path is
  inside the allowlist: write the log, stage, commit, and publish using the
  same success/failure/retry behavior the queue already has.

## Acceptance Criteria

- The only source change is in `Invoke-AiDispatchQueue.ps1`.
- The broad queue staging call is protected by the guard before it can run.
- Hard-coded active handoff artifact matching is constrained to one file
  basename directly under `ai_handoffs/`; it must not use a broad `-like`
  pattern that lets `*` match path separators. For example,
  `ai_handoffs/ISSUE-180_EXEC_2026-05-25_19-00-00+0300.md` may match, but
  `ai_handoffs/ISSUE-180_EXEC_x/nested.md` must not match.
- `.meta.json` sidecars are allowed only when their basename matches an
  allowed active-dispatch TASK, EXEC, or CORRECT packet basename. REVIEW and
  CLOSEOUT packet sidecars are not allowed by the guard.
- The queue log allowance is limited to the exact path returned by
  `Write-DispatchLog` for the current run. A TASK packet token such as
  `ai_dispatch_logs/log_*.md` must not allow additional log files.
- Positive TASK allowlist parsing supports exact file paths, glob path
  patterns, and affirmative directory prefixes. Directory-prefix matching must
  use a path-boundary check.
- The parser does not extract allowed paths from negative sections, forbidden
  sections, halt conditions, constraints, prior-control feedback, or any prose
  outside the positive allowed-path sections.
- The guard validates rename/copy records conservatively: both old and new
  paths must be allowed, not just the destination path.
- On disallowed paths, the queue output names all disallowed changed or
  untracked paths clearly and no `git add -A`, commit, merge, push, or publish
  is attempted for those paths.
- On an all-allowed dispatch, queue staging, commit creation, publish gating,
  labels, comments, and retry behavior remain functionally unchanged.
- The implementation does not add new dependencies, external tools, prompts,
  schemas, configuration files, tests, docs, or workflow changes.

## Constraints / Non-Goals

- Do not change the inner dispatch loop, planner prompts, Claude prompts,
  Codex control schema, handoff protocol, verification script, auto scheduler,
  health reporter, or GitHub issue-selection behavior.
- Do not implement a new sandbox, a full policy language, or a new task packet
  schema.
- Do not loosen queue publish gates. The existing requirement that publish
  occurs only when the loop exits 0 and Codex control verdict is `pass` must
  remain intact.
- Do not broaden allowed dispatch artifacts beyond active `TASK`, `EXEC`, and
  `CORRECT` packets plus matching sidecars.
- Do not rely on unstated trust in `git add -A`; the guard must run before that
  broad staging operation.

## Planner Notes

The prior implementation attempt put the guard in the right queue phase and
passed the canonical verification gate, but Codex control found three blocking
scope bugs. This packet intentionally makes those fixes explicit:

- Hard-coded active-dispatch handoff artifact matching must not use broad
  PowerShell `-like` globbing for active EXEC artifact patterns, because
  `*` can match path separators and allow nested paths.
- `ai_dispatch_logs/` must be controlled by the exact `Write-DispatchLog`
  return path. TASK parsing must not turn `ai_dispatch_logs/log_*.md` into a
  general allowance for extra log files.
- Positive allowed TASK paths may be written as globs or directory prefixes.
  Do not drop bare directory-prefix tokens while parsing the positive sections.

Prefer small helper functions inside `Invoke-AiDispatchQueue.ps1` for path
normalization, active TASK parsing, allowed-path rule construction, and status
entry validation. Keep path comparisons repo-relative with forward slashes so
PowerShell and Git output agree on Windows.
