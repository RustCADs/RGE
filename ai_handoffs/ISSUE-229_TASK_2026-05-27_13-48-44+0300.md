## Goal

Add low-risk mid-run observability to the GitHub issue queue runner for issue
#229. `Invoke-AiDispatchQueue.ps1` should post concise progress comments at
major queue/orchestrator stage boundaries so a human can inspect the GitHub
issue thread and understand where an active dispatch is without reading local
run-dir logs. This is quality-of-life observability only: publish semantics,
retry semantics, terminal labels, queue selection, and the existing final result
comment remain authoritative and unchanged.

## Deliverables

- Add a small, non-terminal progress-comment path in
  `Invoke-AiDispatchQueue.ps1` that posts GitHub issue comments through the
  existing `gh issue comment` style used by the final result comment.
- Add or reuse a pure helper that builds deterministic progress-comment text
  or stage metadata. The helper must be side-effect-free and suitable for
  Pester coverage without invoking `gh`, `git`, `codex`, `claude`, the queue,
  the scheduler, or the network.
- Add a best-effort posting wrapper for non-terminal progress comments. A
  failure to post one of these comments must emit a clear `WARNING:` line and
  continue the dispatch.
- Preserve the existing final result comment, including its core information:
  loop exit code, Codex control verdict, branch/publish outcome, detailed log,
  retry note, loop-output tail, and mode footer.
- Update `AI_DISPATCH_AUTOMATION.md` to document the new progress-comment
  behavior and explicitly state that progress-comment failures are warnings,
  not dispatch failures.
- Add focused Pester regression coverage under `tools/dispatch-tests/**` for
  any new pure helper used to format progress-comment text or stage metadata.

## Acceptance Criteria

- A normal queue run posts progress comments before the existing final result
  comment at these minimum stage boundaries:
  - issue claimed / run starting, including dispatch id and branch name;
  - inner dispatch loop starting, including dispatch id, branch name, and loop
    log path or another stable local log identifier;
  - inner dispatch loop finished, including loop exit code and Codex control
    verdict when available, using `unknown` or an equivalent deterministic
    fallback value when no verdict exists;
  - publish decision starting or skipped, distinguishing automatic publish
    mode from `-NoPublish` branch mode and from not-eligible-to-publish cases.
- Progress comments are concise and deterministic. They identify the issue,
  dispatch id, branch, and local log/audit artifact where available, and they
  do not include full logs, loop-output tails, model transcripts, diffs, or
  control JSON.
- Failure to post any non-terminal progress comment warns and does not fail,
  retry, relabel, publish, or otherwise alter an otherwise healthy dispatch.
- The existing final result comment is still posted after the queue's commit
  and publish decision path with the same core information as before this
  issue.
- `-NoPublish` remains branch-only. A passing run in branch mode may commit to
  the issue branch for human review but must not fast-forward, merge, push, or
  otherwise publish to `origin/main`.
- Auto-publish mode remains unchanged: it is attempted only when the branch was
  committed, the loop exit code is `0`, and the Codex control verdict is
  `pass`.
- Terminal label reconciliation from #227 remains unchanged. The existing
  `Get-DispatchTerminalLabelPlan` behavior, retry eligibility, failure
  taxonomy selection, and post-label verification remain authoritative.
- The docs describe the progress-comment stages and state that progress-comment
  failures are non-terminal warnings, while the final result comment and
  terminal labels remain the durable outcome signals.
- Focused tests cover deterministic formatting/stage metadata for the new pure
  helper and prove that tests do not perform live GitHub calls or run the
  queue.

## Constraints / Non-Goals

- Keep this change additive and low risk. Do not refactor unrelated queue
  behavior.
- Preserve Windows PowerShell 5.1 compatibility.
- Use the existing queue script idioms for temp files, UTF-8 writing,
  `Invoke-Tool`, warnings, and timing traces.
- Do not alter `Invoke-AiDispatchLoop.ps1`; the inner loop should not learn
  about GitHub comments.
- Do not add PR publish mode, separate git worktrees, structured TASK schema
  migrations, retry-policy changes, new failure taxonomy labels, selector
  changes, or scheduler changes.
- Do not make progress comments a gate for publishing, labels, retry, or issue
  closure.
- Do not dump large logs or model output into progress comments. The existing
  final result comment remains the only comment that includes the loop-output
  tail.

  formatter can be tested by dot-sourcing `Invoke-AiDispatchQueue.ps1` through
  the existing `RGE_AI_DISPATCH_QUEUE_SKIP_MAIN` seam.
- Stage comments should be short status markers, not audit logs. Prefer stable
  fields such as issue number, dispatch id, branch, loop log path, detailed
  queue log path when already known, loop exit code, control verdict, and mode.

  durable outcome records. Progress comments only make an active run easier to
  follow while it is happening.
